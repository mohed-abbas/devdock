## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Forced `--no-cache` rebuild of `devdock-app` / `devdock-terminal` images**

- **Found during:** Task 2 Step 5 (first stack-smoke cold run)
- **Issue:** Stack-smoke reported `container devdock-terminal is unhealthy`. Investigation showed the running containers lacked `wget`, `gosu`, and `postgresql-client` despite the Dockerfile explicitly `apt-get install`ing them at lines 54-59 (app-runner) and 106-110 (terminal-runner). The pre-existing images were dated 2026-04-18T22:29:11 (earlier than the first Dockerfile commit `634423f` at 23:52:52 same day) — stale artifacts from an earlier dev-environment build. `docker compose up --build` reused cached layers and reported "Built" without actually running the apt-install steps.
- **Fix:** `docker image rm devdock-app:latest devdock-terminal:latest` + `docker compose -f docker-compose.yml build --no-cache`. Post-rebuild verification: `docker run --rm --entrypoint="" devdock-app which wget gosu pg_isready` returned `/usr/bin/wget /usr/sbin/gosu /usr/bin/pg_isready`.
- **Files modified:** None (image-only action; no git-tracked file changes)
- **Commit:** N/A — not a code change

**2. [Rule 3 — Blocking] Added `command:` override with absolute tsx path to terminal service**

- **Found during:** Task 2 Step 5 (first stack-smoke run after `--no-cache` rebuild)
- **Issue:** Terminal container entered a restart loop: `error: exec: "tsx": executable file not found in $PATH`. Root cause: `Dockerfile` terminal-runner stage sets `CMD ["tsx", "server/terminal-server.ts"]`, but `tsx` lives at `/app/node_modules/.bin/tsx` which is NOT on the base `node:22-slim` PATH. The entrypoint's `exec gosu termuser "$@"` then fails bare-name lookup via execve. Pre-existing Dockerfile/CMD mismatch that was masked by the GAP-5 loadConfig crash happening earlier in boot.
- **Fix:** Added `command: ["/app/node_modules/.bin/tsx", "server/terminal-server.ts"]` to the terminal service in `docker-compose.yml`. This overrides the Dockerfile CMD with the absolute binary path, bypassing PATH lookup entirely. Scope stays inside docker-compose.yml (already in `files_modified`).
- **Files modified:** `docker-compose.yml`
- **Commit:** `86dca96`

**3. [Rule 3 — Blocking] Removed `GITHUB_*` empty-string pass-throughs from app env**

- **Found during:** Task 2 Step 5 (first stack-smoke run after deviation 2 fix)
- **Issue:** App container unhealthy with `Invalid environment variables: { GITHUB_CLIENT_ID: [ 'String must contain at least 1 character(s)' ], GITHUB_CLIENT_SECRET: [ ... ], GITHUB_TOKEN_ENCRYPTION_KEY: [ 'Must be 64 hex chars...' ] }`. Root cause: `docker-compose.yml` used `GITHUB_CLIENT_ID: ${GITHUB_CLIENT_ID:-}` etc., which sets the container var to empty string when unset on host. `src/lib/config.ts` envSchema declares the three as `.min(1).optional()` / `.regex().optional()` without `.default('')` — zod rejects empty strings even under `.optional()`, because empty string is a defined value that fails the min/regex predicate.
- **Fix:** Removed the three `GITHUB_*` lines from the app `environment:` block (replaced with an explanatory comment). When the host has no `GITHUB_*` env var set, compose now does NOT add the key to the container's environment at all; zod sees `undefined` and `.optional()` passes.
- **Files modified:** `docker-compose.yml`
- **Commit:** `86dca96` (same commit as deviation 2; one atomic compose-side cascade-fix commit)
- **Scope preservation:** No source code touched. `src/lib/config.ts` envSchema unchanged (D-07 preserved). If a future phase wants tighter defensive behavior (e.g., `.default('').transform(undefinedIfEmpty)`), that's a source-code change and belongs in its own plan.
