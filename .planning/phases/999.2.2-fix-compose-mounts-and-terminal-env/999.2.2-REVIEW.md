## Summary

Phase 999.2.2 wired `DATABASE_URL` into the terminal service and converted three host-path env-template entries to a documented pattern. The changes are correct and preserve the D-09 mirror-mount invariant. No critical issues were found.

Two warnings cover defence-in-depth gaps around `POSTGRES_PASSWORD` expansion in `DATABASE_URL` — both the app and terminal services expand the variable without a guard or URL-encoding, which relies on the postgres service's `:?` guard to fail-fast and on the operator choosing a URL-safe password. Four info items note small documentation/consistency nits in `.env.example`.

The terminal service `DATABASE_URL` line (docker-compose.yml:155) is byte-identical to the app service's line (docker-compose.yml:88), which is the intended outcome for the phase. The mirror-mount pattern `${VAR}:${VAR}` is intact on all three bind mounts (lines 118–120, 165).

## Warnings

### WR-01: DATABASE_URL silently expands empty password if POSTGRES_PASSWORD is unset

**File:** `docker-compose.yml:88`, `docker-compose.yml:155`
**Issue:** Both `DATABASE_URL` lines interpolate `${POSTGRES_PASSWORD}` without a default or `:?` guard. If `POSTGRES_PASSWORD` is unset, compose expands to `postgresql://devdock:@postgres:5432/devdock` (empty password). Today this is masked because the `postgres` service on line 24 uses `${POSTGRES_PASSWORD:?...}` and aborts the whole stack before `app`/`terminal` start, so the malformed URL is never actually used. Relying on one service's guard to protect another is fragile — if someone later removes the postgres guard, or runs `docker compose run app` without postgres, the app will silently connect (or attempt to) with a blank password.
**Fix:** Add the same `:?` guard here for defence-in-depth, so the URL itself fail-fasts:
```yaml
# app service
DATABASE_URL: postgresql://${POSTGRES_USER:-devdock}:${POSTGRES_PASSWORD:?POSTGRES_PASSWORD must be set in .env.local}@postgres:5432/${POSTGRES_DB:-devdock}
# terminal service — same change
DATABASE_URL: postgresql://${POSTGRES_USER:-devdock}:${POSTGRES_PASSWORD:?POSTGRES_PASSWORD must be set in .env.local}@postgres:5432/${POSTGRES_DB:-devdock}
```

### WR-02: POSTGRES_PASSWORD is not URL-encoded when embedded in DATABASE_URL

**File:** `docker-compose.yml:88`, `docker-compose.yml:155`, `.env.example:81`
**Issue:** `DATABASE_URL` embeds `${POSTGRES_PASSWORD}` verbatim. A password containing any of `@ : / ? # % &` will produce a malformed URL that drivers will either misparse (e.g. `p@ss` → userinfo `devdock:p`, host `ss@postgres`) or reject outright. `.env.example` instructs operators to `change-me-in-env-local` without warning about URL-safe characters, so it is plausible that a strong random password generated by e.g. `pwgen -s 32 1` will contain `/` or `+`.
**Fix:** Document the constraint in `.env.example` near the `POSTGRES_PASSWORD` line, or generate the password with a URL-safe alphabet:
```bash
# .env.example, near POSTGRES_PASSWORD:
# POSTGRES_PASSWORD is embedded verbatim in DATABASE_URL. Use only URL-safe
# characters [A-Za-z0-9_-]. Generate with: openssl rand -hex 32
POSTGRES_PASSWORD=change-me-in-env-local
```
A more robust fix is to have the application read user/password/host/db as separate env vars and build the URL in code with proper encoding, but that is out of scope for this phase.

## Info

### IN-01: Commented-out DATABASE_URL override in .env.example is a foot-gun

**File:** `.env.example:87`
**Issue:** Line 87 provides a commented-out override `DATABASE_URL=postgresql://devdock:change-me-in-env-local@postgres:5432/devdock`. In the current compose setup, `app` and `terminal` synthesize their own `DATABASE_URL` from `POSTGRES_USER`/`POSTGRES_PASSWORD`/`POSTGRES_DB` (docker-compose.yml:88, 155), so this env-level override no longer takes effect inside the containers — but it *will* be picked up by host-side tools (drizzle migrations, `psql`-based scripts, `npm run terminal:dev`). A user who uncomments this and forgets to update `POSTGRES_PASSWORD` ends up with the host-side tooling using one password and the container using another.
**Fix:** Either delete the commented override (compose now synthesizes it), or rewrite the comment to make the split-brain explicit:
```bash
# Override DATABASE_URL for HOST-SIDE tooling (drizzle migrations, local `npm run
# terminal:dev`) when running against the compose-hosted postgres. The compose
# `app` and `terminal` services synthesize DATABASE_URL internally from the
# POSTGRES_* vars below and ignore this line. Keep in sync with POSTGRES_PASSWORD.
# DATABASE_URL=postgresql://devdock:change-me-in-env-local@localhost:5432/devdock
```
(Note: the commented example also uses host `postgres` which only resolves inside the compose network — a host-side tool cannot reach `postgres:5432`. Changing the example to `localhost:5432` matches the top-of-file default.)

### IN-02: Maintainer's personal username leaks into .env.example

**File:** `.env.example:76`
**Issue:** The example path `/home/murx` is the maintainer's actual Linux username. Line 42 (`CLAUDE_CONFIG_PATH`) uses the placeholder `/home/you/.claude`, which is the convention for the rest of the file. Inconsistent, and mildly confusing for other operators.
**Fix:**
```bash
# PRODUCTION_APPS_DIR=/home/you/apps
```

### IN-03: Env vs mount default asymmetry for optional host-path vars is undocumented in compose

**File:** `docker-compose.yml:101-102`, `docker-compose.yml:119`, `docker-compose.yml:98`, `docker-compose.yml:120`, `docker-compose.yml:162`, `docker-compose.yml:165`
**Issue:** For `PRODUCTION_APPS_DIR` and `CLAUDE_CONFIG_PATH`, the container env uses `${VAR:-}` (empty default) while the bind mount uses `${VAR:-/srv/devdock/...-empty}` (sentinel path). This is an intentional pattern: container code checks the env as a "configured?" flag, while the mount always succeeds against a sentinel empty directory. Correct, but a maintainer reading only the env block (or only the volume block) will not see the full picture. `DEVDOCK_DATA_DIR` uses the sentinel default on *both* sides (line 96 + 118), which adds to the asymmetry.
**Fix:** Add a short comment next to each env-with-empty-default line cross-referencing the mount:
```yaml
# Empty when unconfigured so code can branch on it. The mount below uses a
# sentinel empty dir so the bind mount is always valid regardless.
PRODUCTION_APPS_DIR: ${PRODUCTION_APPS_DIR:-}
```

### IN-04: Phase spec said "blank-value" for all three host-path vars; .env.example uses commented-out example for two of them

**File:** `.env.example:29`, `.env.example:42`, `.env.example:76`
**Issue:** The phase brief stated that `DEVDOCK_DATA_DIR`, `CLAUDE_CONFIG_PATH`, and `PRODUCTION_APPS_DIR` were converted to "blank-value + absolute-path comment blocks". In the file, only `DEVDOCK_DATA_DIR=` (line 29) is actually blank-valued; `CLAUDE_CONFIG_PATH` (line 42) and `PRODUCTION_APPS_DIR` (line 76) are commented-out example assignments (`# CLAUDE_CONFIG_PATH=/home/you/.claude`). Functionally equivalent (both resolve to "unset" at compose time and trigger the `:-` sentinel default), and the surrounding comment block does say "Leave commented to fall back to...", so the distinction is deliberate — but it diverges from the phase brief's wording. If the brief is authoritative, these two should become blank assignments; if the file is authoritative, the brief's summary should be corrected for the plan record.
**Fix:** No change needed if the current file form is intentional. Otherwise:
```bash
# Was: # CLAUDE_CONFIG_PATH=/home/you/.claude
CLAUDE_CONFIG_PATH=

# Was: # PRODUCTION_APPS_DIR=/home/murx
PRODUCTION_APPS_DIR=
```
