# Phase 4: Web Terminal & Claude Code - Research

**Researched:** 2026-04-13
**Domain:** Browser terminal (xterm.js), WebSocket transport (Socket.IO), Docker exec integration
**Confidence:** MEDIUM

## Summary

This phase delivers a browser-based terminal connected to running Docker environments via xterm.js + Socket.IO + dockerode exec. The primary technical challenge is integrating Socket.IO with the existing Next.js App Router application that uses `output: 'standalone'`. Next.js custom servers are incompatible with standalone output, so Socket.IO must run as a separate process on a different port, proxied through nginx alongside the existing Next.js app.

The terminal data flow is: xterm.js (browser) -> Socket.IO client -> Socket.IO server (separate process) -> dockerode exec stream -> container PTY. Resize events flow the same path. Authentication uses short-lived tokens generated by the Next.js API and validated by the Socket.IO server.

**Primary recommendation:** Run Socket.IO as a standalone Node.js/Express process on a separate port (e.g., 3001), proxied by the existing nginx config at a sub-path like `/ws/`. This preserves Next.js `output: 'standalone'` and keeps the WebSocket server simple and debuggable. Use xterm.js 5.5.0 (not 6.0.0) for addon compatibility stability.

## Architecture Patterns

### Critical: Socket.IO + Next.js Standalone

**Problem:** The project uses `output: 'standalone'` in `next.config.ts`. Next.js custom servers are incompatible with standalone output. [VERIFIED: Next.js docs, GitHub discussions] Socket.IO requires attaching to an HTTP server, which normally means a custom server.

**Solution:** Run Socket.IO as a **separate Node.js process** alongside the Next.js app. Two processes, one nginx proxy.

```
Browser
  |
  +-- HTTP (pages, API) --> nginx :443 --> Next.js :3000
  |
  +-- WebSocket (/ws/) --> nginx :443 --> Socket.IO :3001
```

**Why this works:**
1. Next.js keeps `output: 'standalone'` -- no deployment changes
2. Socket.IO server is a simple ~100-line Express app -- easy to debug
3. nginx already proxies Next.js; adding a WebSocket location block is trivial
4. Both processes share the same Docker socket and database connection
5. systemd manages both processes (two service files, or a single service with ExecStartPre)

### Pattern 1: Docker Exec with PTY

**What:** Create an interactive shell session inside a running container using dockerode exec API with TTY enabled.

**When to use:** Every time a terminal tab is opened.

**Example:**
```typescript
// Source: dockerode README + Docker API docs [VERIFIED: dockerode GitHub]
import Docker from 'dockerode';

const docker = new Docker({ socketPath: '/var/run/docker.sock' });

async function createExecSession(containerId: string, cols: number, rows: number) {
  const container = docker.getContainer(containerId);

  const exec = await container.exec({
    Cmd: ['/bin/bash'],
    AttachStdin: true,
    AttachStdout: true,
    AttachStderr: true,
    Tty: true,           // Allocate PTY -- critical for interactive terminal
    User: 'dev',         // D-03: run as dev user
    Env: ['TERM=xterm-256color'],
  });

  const stream = await exec.start({
    hijack: true,        // HTTP connection hijacking for bidirectional stream
    stdin: true,
    Tty: true,
  });

  // With Tty: true, stdout and stderr are multiplexed into a single stream
  // No need for docker.modem.demuxStream()

  return { exec, stream };
}

// Resize the PTY when terminal dimensions change
async function resizeExec(execId: string, cols: number, rows: number) {
  const exec = docker.getExec(execId);
  await exec.resize({ h: rows, w: cols });
}
```

### Pattern 2: Socket.IO Namespace for Terminal Sessions

**What:** Use a Socket.IO namespace to isolate terminal communication from other potential real-time features.

**Example:**
```typescript
// Source: Socket.IO docs [CITED: socket.io/docs/v4/namespaces]
import { Server } from 'socket.io';

const io = new Server(httpServer, {
  cors: { origin: process.env.NEXTAUTH_URL },
  path: '/ws/socket.io',
});

const terminalNs = io.of('/terminal');

terminalNs.use(async (socket, next) => {
  // Validate short-lived token from query params (D-05)
  const token = socket.handshake.auth?.token || socket.handshake.query?.token;
  if (!token) return next(new Error('Authentication required'));

  const session = await validateTerminalToken(token);
  if (!session) return next(new Error('Invalid or expired token'));

  socket.data.environmentId = session.environmentId;
  socket.data.userId = session.userId;
  next();
});

terminalNs.on('connection', (socket) => {
  socket.on('exec:create', async ({ cols, rows }) => {
    // Create Docker exec session, pipe stream to socket

  socket.on('exec:input', (data: string) => {
    // Write to exec stream stdin
  });

  socket.on('exec:resize', ({ cols, rows }) => {
    // Resize PTY
  });

  socket.on('disconnect', () => {
    // Clean up exec sessions
  });
});
```

### Pattern 3: Short-Lived Terminal Token

**What:** Next.js API generates a token that the browser passes to Socket.IO for authentication. Avoids exposing JWT session directly to WebSocket.

**Example:**
```typescript
// API route: src/app/api/terminal/token/route.ts
import { nanoid } from 'nanoid';

// In-memory token store with TTL (acceptable for single-server)
const tokenStore = new Map<string, { environmentId: string; userId: string; expiresAt: number }>();

export async function POST(request: Request) {
  const session = await auth(); // Auth.js session
  if (!session?.user?.id) return Response.json({ error: 'Unauthorized' }, { status: 401 });

  const { environmentId } = await request.json();
  // Validate environment exists, is running, belongs to user...

  const token = nanoid(32);
  tokenStore.set(token, {
    environmentId,
    userId: session.user.id,
    expiresAt: Date.now() + 30_000, // 30 second TTL
  });

  return Response.json({ token });
}
```

### Anti-Patterns to Avoid

- **Custom Next.js server:** Incompatible with `output: 'standalone'`. Would break the existing production deployment pattern. [VERIFIED: Next.js docs]
- **Polling-based terminal I/O:** Never poll for terminal output. WebSocket/stream is the only viable approach for interactive terminals.
- **Mounting Docker socket into containers:** Explicitly forbidden (INFRA-04). Terminal exec happens server-side through dockerode.
- **Exposing JWT session token to WebSocket:** Use short-lived single-use tokens instead. JWT tokens are long-lived and should stay in HTTP-only cookies.
- **Using `exec` (child_process) for Docker exec:** Use dockerode's exec API for stream-based bidirectional communication. The `execFile` pattern is for one-shot compose commands.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Terminal emulation | VT100 parser | xterm.js | Thousands of escape sequences, Unicode, ligatures, selection |
| WebSocket reconnect | Custom retry logic | Socket.IO auto-reconnect | Exponential backoff, transport fallback, connection state machine |
| Terminal resize | Manual dimension calculation | addon-fit + ResizeObserver | Handles font metrics, scrollbar width, DPR |
| Copy/paste in terminal | Clipboard API wrapper | addon-clipboard | Cross-browser compat, OSC 52, selection handling |
| Docker exec streams | Raw HTTP hijack | dockerode exec API | Stream multiplexing, error handling, connection lifecycle |

**Key insight:** Terminal emulation is a deeply complex domain. xterm.js handles thousands of edge cases in character rendering, escape sequences, and input handling. The addons handle the equally complex resize math and clipboard integration. Focus implementation effort on the Socket.IO <-> dockerode bridge and the UI shell.

## Common Pitfalls

### Pitfall 1: TTY Stream Multiplexing Confusion
**What goes wrong:** With `Tty: false`, Docker multiplexes stdout and stderr into a single stream with 8-byte headers. With `Tty: true`, the stream is raw (no headers). Using `demuxStream()` on a TTY stream corrupts output.
**Why it happens:** Docker API behaves differently based on TTY allocation.
**How to avoid:** Always use `Tty: true` for interactive terminals. Pipe the raw stream directly to Socket.IO without demultiplexing.
**Warning signs:** Garbled output, missing characters, binary data appearing in terminal. [CITED: Docker API docs, dockerode README]

### Pitfall 2: Exec Session Cleanup on Disconnect
**What goes wrong:** Zombie exec processes accumulate inside containers when WebSocket disconnects without cleanup.
**Why it happens:** Docker exec sessions persist until the process exits. If the Socket.IO connection drops, the bash process keeps running.
**How to avoid:** On socket disconnect, destroy the exec stream (which sends SIGHUP to the process). Track exec sessions per socket and clean up on disconnect. Consider a periodic inspection of running exec sessions.
**Warning signs:** `docker top <container>` shows many bash processes. [ASSUMED]

### Pitfall 3: Resize Race Condition
**What goes wrong:** Terminal displays garbled output after resize because the resize event reaches Docker before/after xterm.js updates.
**Why it happens:** FitAddon calculates new dimensions, sends resize to server, and xterm.js reflows content -- these happen asynchronously.
**How to avoid:** Debounce resize events (100-150ms). Send resize to server only after FitAddon has recalculated. Apply resize server-side via `exec.resize()` which updates the PTY.
**Warning signs:** Wrapping issues, truncated lines after browser window resize. [ASSUMED]

### Pitfall 4: Socket.IO Transport Mismatch with nginx
**What goes wrong:** Socket.IO connection fails or falls back to long-polling permanently.
**Why it happens:** nginx must be configured for WebSocket upgrade (`Upgrade` and `Connection` headers). Default nginx config does not proxy WebSocket.
**How to avoid:** Use the nginx WebSocket proxy config pattern (see Architecture Patterns above). Set `proxy_read_timeout` to a high value to prevent idle WebSocket disconnection.
**Warning signs:** Socket.IO client shows `transport: polling` instead of `websocket` in debug logs. [VERIFIED: Socket.IO docs]

### Pitfall 5: ~/.claude Mount Exposes Credentials
**What goes wrong:** The `.credentials.json` file in `~/.claude/` contains Claude authentication tokens. Mounting the full directory exposes these to container users.
**Why it happens:** D-07 specifies "Full `~/.claude` directory mounted read-only" but the directory contains sensitive files (`.credentials.json`, `sessions/`, `history.jsonl`).
**How to avoid:** The ANTHROPIC_API_KEY is separately injected via D-08 (environment variable). The `.credentials.json` is the *host's* Claude credentials. Since containers are read-only mounted AND the user controls both the host and containers (single-user self-hosted tool), this is acceptable per the locked decision. However, document this clearly: the mount gives container processes read access to host Claude credentials.
**Warning signs:** N/A -- this is a deliberate design choice. [VERIFIED: ls ~/.claude/ shows .credentials.json]

### Pitfall 6: xterm.js Import in SSR
**What goes wrong:** Next.js server-side rendering crashes because xterm.js requires browser APIs (DOM, window).
**Why it happens:** xterm.js is a browser-only library. Next.js App Router renders components on the server by default.
**How to avoid:** Use `'use client'` directive on all terminal components. Use `dynamic(() => import(...), { ssr: false })` for the terminal instance component to prevent any server-side import.
**Warning signs:** `ReferenceError: window is not defined` during build or SSR. [VERIFIED: Socket.IO Next.js docs]

### Pitfall 7: Container Name Resolution for Exec
**What goes wrong:** Cannot find the right container to exec into because the container name doesn't match expectations.
**Why it happens:** Docker Compose names containers as `devdock-{slug}-dev-1` (with a numeric suffix). The `getProjectStatus()` function already lists containers with labels.
**How to avoid:** Use the `com.docker.compose.service=dev` label filter to find the dev container for exec, not the container name. The existing `getProjectStatus()` function returns container IDs which can be used directly with dockerode.
**Warning signs:** "Container not found" errors when trying to open terminal. [VERIFIED: docker-service.ts code]

```typescript
// Source: Socket.IO docs [CITED: socket.io/docs/v4/client-initialization]
import { io, Socket } from 'socket.io-client';

async function connectTerminal(environmentId: string): Promise<Socket> {
  // Step 1: Get short-lived token from Next.js API
  const res = await fetch('/api/terminal/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ environmentId }),
  });
  const { token } = await res.json();

  // Step 2: Connect Socket.IO with token
  const socket = io('/terminal', {
    path: '/ws/socket.io',
    auth: { token },
    reconnection: true,
    reconnectionAttempts: 10,
    reconnectionDelay: 1000,
    reconnectionDelayMax: 5000,
  });

  return socket;
}
```

## Open Questions

1. **Token store sharing between Next.js and Socket.IO processes**
   - What we know: Next.js API generates tokens, Socket.IO server validates them. Two separate Node.js processes.
   - What's unclear: How to share the in-memory token store across processes.
   - Recommendation: Use the PostgreSQL database (already shared) for token storage. A `terminal_tokens` table with `token`, `environment_id`, `user_id`, `expires_at` columns. Tokens are short-lived (30s) and can be cleaned up periodically. Alternatively, use a signed JWT-style token that both processes can verify using a shared secret (the `AUTH_SECRET` env var).

2. **systemd service configuration for two processes**
   - What we know: The project uses systemd for process management.
   - What's unclear: Whether to use two service files or one with multiple ExecStart.
   - Recommendation: Two service files (`devdock.service` and `devdock-terminal.service`) with `devdock-terminal.service` depending on `devdock.service` via `After=devdock.service`. Simpler to manage, restart independently.

3. **ANTHROPIC_API_KEY storage location**
   - What we know: D-08 says "user sets it once in DevDock settings."

   - Recommendation: Add a `settings` table or store in environment variables. For v1, an env var in `.env.local` is simplest. A settings table (key-value) can come with Phase 6 dashboard polish.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Short-lived token for WebSocket auth (D-05). Auth.js session validates before token generation. |
| V3 Session Management | yes | Terminal tokens expire in 30s. Socket.IO sessions tied to authenticated user. |
| V4 Access Control | yes | Verify environment belongs to authenticated user before allowing exec. |
| V5 Input Validation | yes | Validate environment ID (UUID format), terminal dimensions (integer bounds). zod schemas. |
| V6 Cryptography | no | No crypto operations in this phase. Token is random (nanoid), not encrypted. |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| WebSocket hijacking | Spoofing | Short-lived token auth, token consumed on use (single-use) |
| Container escape via exec | Elevation | Docker socket not in container (INFRA-04), exec runs as `dev` user (not root) |
| Terminal command injection | Tampering | No server-side command construction from user input. Exec runs `/bin/bash` only. User types freely in their own container. |
| Token replay | Spoofing | 30s TTL, single-use tokens, tokens scoped to specific environment |
| Cross-user container access | Information Disclosure | Validate environment ownership before issuing terminal token |
| DoS via many exec sessions | Denial of Service | Max 5 tabs per environment (UI-SPEC), cleanup on disconnect |
