**Step 1 -- Create `src/components/auth/login-form.tsx`** (client component per UI-SPEC):

This is a 'use client' component using React 19's `useActionState` hook (NOT `useFormState` which is deprecated).

```typescript
'use client';

import { useActionState, useRef, useEffect } from 'react';
import { login } from '@/lib/auth/actions';
import { Button } from '@/components/ui/button';
import { Input } from '@/components/ui/input';
import { Label } from '@/components/ui/label';
import { Card, CardContent } from '@/components/ui/card';

export function LoginForm() {
  const [state, formAction, isPending] = useActionState(login, { error: null });
  const usernameRef = useRef<HTMLInputElement>(null);

  // After error, focus returns to username input (UI-SPEC accessibility)
  useEffect(() => {
    if (state.error && usernameRef.current) {
      usernameRef.current.focus();
    }
  }, [state.error]);

  return (
    <Card className="w-full">
      <CardContent className="pt-6">
        <form action={formAction} className="space-y-4">
          <div className="space-y-2">
            <Label htmlFor="username">Username</Label>
            <Input
              id="username"
              name="username"
              type="text"
              required
              autoComplete="username"
              autoFocus
              ref={usernameRef}
            />
          </div>
          <div className="space-y-2">
            <Label htmlFor="password">Password</Label>
            <Input
              id="password"
              name="password"
              type="password"
              required
              autoComplete="current-password"
            />
          </div>
          {/* Fixed-height error region to prevent layout shift (UI-SPEC) */}
          <div className="min-h-[20px]" aria-live="polite" aria-atomic="true">
            {state.error && (
              <p className="text-sm text-destructive">{state.error}</p>
            )}
          </div>
          <Button
            type="submit"
            className="w-full h-11"
            disabled={isPending}
          >
            {isPending ? 'Signing in...' : 'Sign In'}
          </Button>
        </form>
      </CardContent>
    </Card>
  );
}
```

Key UI-SPEC compliance points:
- No CardHeader used (UI-SPEC says "no CardHeader, use CardContent only")
- `pt-6` on CardContent for top padding
- `space-y-4` for form field gap (16px)
- `space-y-2` for label-to-input gap (8px)
- `min-h-[20px]` error region with `aria-live="polite"` to prevent layout shift
- `text-sm text-destructive` for error text
- Button `w-full h-11` (full width, 44px touch target per UI-SPEC)
- `autoFocus` on username input
- `autoComplete="username"` and `autoComplete="current-password"` for accessibility
- No placeholder text on inputs (UI-SPEC: "none -- leave empty, label is sufficient")
- No "Remember me", "Forgot password", "Sign up", or social login (UI-SPEC negative list)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Login form -> server action | Username and password from untrusted client input to server-side action |
| Client state -> UI | Error messages from server action displayed in client component |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Information Disclosure | login-form.tsx | mitigate | Error messages are generic per D-04 ("Invalid username or password."); never field-specific; no stack traces shown to user |
| T-02-08 | Spoofing | login page | mitigate | Form submits via server action (not client fetch) so Auth.js CSRF protection applies; autoComplete attributes help password managers |
| T-02-09 | Tampering | form inputs | accept | Client-side required attributes are UX only; server-side validation in authorize() is the real gate; low risk since server action validates all inputs |
</threat_model>
