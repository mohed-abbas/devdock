## New Gaps Surfaced (NOT in original 999.2.1 scope)

These were masked behind GAPs 1+2. Each one became visible only after the preceding fix unblocked the next stage of the gate.

### GAP-3 — `Caddyfile` `persist_config on` invalid ✅ FIXED inline
- **Symptom:** Caddy container fail-loop: `Error: adapting config using caddyfile: parsing caddyfile tokens for 'persist_config': persist_config must be 'off'`
- **Root cause:** The Caddyfile global option `persist_config` is a disable-switch only — it accepts `off` or nothing. Default behavior is persist-on, which matches the author's stated intent in both the commit message (`edc0e20`) and inline comment.
- **Fix:** Removed `persist_config on` (commit `1878cf2`). Persistence now enabled via Caddy's default behavior; comment updated to reflect the correct semantics.
- **Validation:** `docker run --rm -v .../Caddyfile:/etc/caddy/Caddyfile caddy:2.9-alpine caddy validate` → `Valid configuration`.

### GAP-4 — Compose volume bind-mounts hard-coded to `/srv/devdock/*` ⚠️ local-worked-around, NOT committed
- **Symptom:** `docker compose up` fails: `mounts denied: The path /srv/devdock/data is not shared from the host`
- **Root cause:** `docker-compose.yml` defaults `DEVDOCK_DATA_DIR`, `PRODUCTION_APPS_DIR`, `CLAUDE_CONFIG_PATH` to `/srv/devdock/*` (production VPS paths). `.env.example` has only `DEVDOCK_DATA_DIR=./data` (relative) as guidance; the user's `.env` was missing all three. And relative paths break because the mount spec uses the same var for source and target (`${VAR}:${VAR}`), and container mount paths must be absolute.
- **Local workaround:** Set absolute paths in `.env` (uncommitted; `.env` is gitignored) and created empty local dirs:
  ```
  DEVDOCK_DATA_DIR=/Users/murx/developer/personal/devdock/data
  PRODUCTION_APPS_DIR=/Users/murx/developer/personal/devdock/data/apps-empty
  CLAUDE_CONFIG_PATH=/Users/murx/developer/personal/devdock/data/claude-empty
  ```
- **Root fix scope for 999.2.2:** Decide policy — update `.env.example` to document the absolute-path requirement for local dev, or change the compose `VAR:VAR` mount pattern to `${VAR}:/some/fixed/container/path` so the container path is always absolute and the host path can be relative in dev.

### GAP-5 — `terminal` service missing `DATABASE_URL` env ❌ UNFIXED
- **Symptom:** `devdock-terminal` container loops `unhealthy`:
  ```
  Invalid environment variables: { DATABASE_URL: [ 'Required' ] }
  Error: Invalid environment configuration
      at loadConfig (/app/src/lib/config.ts:45:11)
  ```
- **Root cause:** `docker-compose.yml:145-153` (terminal service environment block) does NOT set `DATABASE_URL`. But `server/terminal-server.ts` imports from `src/lib/config.ts`, which calls `loadConfig()` at module-init time and hard-fails on missing `DATABASE_URL`. The `app` service (line 88) does set it; the terminal service was likely overlooked.
- **Fix scope for 999.2.2:** Either (a) add `DATABASE_URL: postgresql://${POSTGRES_USER:-devdock}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB:-devdock}` to the terminal service env block, or (b) restructure `src/lib/config.ts` so terminal-only paths don't require `DATABASE_URL` validation. Option (a) is smaller; option (b) is cleaner separation of concerns.
