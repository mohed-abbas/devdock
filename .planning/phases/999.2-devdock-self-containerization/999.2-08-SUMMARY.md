## What Was Built

### `src/lib/docker/caddy-lifecycle.ts` (new file)

Higher-level bridge between environment DB records and the Caddy Admin API HTTP client. Exports two functions:

- `registerPreviewRoute(env: { id, slug, previewPort })` — gates on `config.PREVIEW_DOMAIN` truthiness and `env.previewPort` non-null; swallows Caddy errors with `console.error`; logs success with `console.log`.
- `deregisterPreviewRoute(envSlug: string)` — gates on `config.PREVIEW_DOMAIN`; swallows errors; idempotent (safe if route was never registered).

Both functions catch all errors internally — callers never need a try/catch.

**Why in `caddy-lifecycle.ts` and not `caddy-admin.ts`:** Plan 06 committed to `caddy-admin.ts` being a pure HTTP client with no dependency on `config.ts` or DB record shapes. Adding environment-aware logic there would violate that contract. `caddy-lifecycle.ts` absorbs the domain bridge concern cleanly.

## Graceful Degradation Behavior

All Caddy Admin API call paths:

1. `registerPreviewRoute` → `addPreviewRoute` (HTTP POST to Caddy) — exception caught in `caddy-lifecycle.ts`, logged, never rethrown
2. `deregisterPreviewRoute` → `removePreviewRoute` (HTTP DELETE to Caddy) — same pattern

Environment create/start/stop/delete operations succeed even if Caddy is unreachable, misconfigured, or returns an error. This is intentional per plan success criteria: Caddy failures are non-fatal. Re-registration happens on the next environment start.
