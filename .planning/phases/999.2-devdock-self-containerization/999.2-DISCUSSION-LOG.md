## Proxy & Host Coexistence

### Q1 — Reverse proxy inside DevDock's stack

| Option | Description | Selected |
|--------|-------------|----------|
| Caddy | Auto-TLS, simpler config, container-name routing. Matches locked direction. | ✓ |
| Keep nginx (containerized) | Reuse existing deploy/nginx configs inside a container. Manual cert wiring. | |

**User's choice:** Caddy

### Q2 — Coexistence with host nginx (port 80/443 ownership)

| Option | Description | Selected |
|--------|-------------|----------|
| Host nginx stays in front | Host owns :80/:443 and TLS; proxies devdock.<domain> + *.preview.devdock.<domain> to DevDock Caddy on internal port. | ✓ |
| DevDock Caddy owns :80/:443 | Caddy replaces host nginx; prod app routing must move into Caddy. | |
| Split ports (Caddy on :8443) | Caddy publishes on non-standard port; ugly URLs. | |

**User's choice:** Host nginx stays in front
**Notes:** Preserves existing certbot workflow and keeps production routing risk-free.

### Q3 — Wildcard preview cert acquisition

| Option | Description | Selected |
|--------|-------------|----------|
| Host certbot DNS-01 kept + HTTP-only Caddy | Existing certbot keeps certs; DevDock Caddy runs HTTP-only; no DNS plugin in stack. | ✓ |
| Caddy auto-DNS-01 inside stack | Caddy acquires wildcard cert via DNS provider plugin (Cloudflare, etc.). | |

**User's choice:** HTTP-only Caddy, host owns TLS
**Notes:** User asked where Caddy DNS-01 would live — clarified it would run inside the containerized Caddy service. After clarification, user selected the simpler host-owned-TLS path because host certbot already has a working wildcard cert workflow.

### Q4 — Production traffic (/home/murx/apps)

| Option | Description | Selected |
|--------|-------------|----------|
| Untouched — host nginx keeps owning it | Existing /home/murx/shared/nginx configs stay as-is. | ✓ |
| Read-only mount into DevDock's proxy | DevDock proxy reads host nginx configs via RO volume. | |

**User's choice:** Untouched

## Service Decomposition

### Q1 — App + terminal service layout

| Option | Description | Selected |
|--------|-------------|----------|
| Two services: app + terminal | Matches current split; independent restart; Socket.IO resource isolation. | ✓ |
| Merge: one Next.js custom server | Single container hosting HTTP + Socket.IO; non-trivial refactor. | |
| Three services: app + terminal + migrator | Adds short-lived migrator; cleaner startup, one more service to reason about. | |

**User's choice:** Two services

### Q2 — Docker socket access for app

| Option | Description | Selected |
|--------|-------------|----------|
| Bind-mount /var/run/docker.sock | App as non-root with docker group gid. Simplest, matches current pattern. | ✓ |
| tecnativa/docker-socket-proxy sidecar | Scoped socket permissions; better least-privilege. Extra service to audit. | |
| Docker-in-Docker | Isolated daemon; breaks 'manage host containers' model. | |

**User's choice:** Bind-mount socket directly

### Q3 — App Dockerfile strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Multi-stage Dockerfile → standalone prod; bind-mount src in dev | Single Dockerfile; docker-compose.override.yml enables HMR in dev. | ✓ |
| Prebuilt standalone only, no dev HMR in-container | Host npm scripts in dev; dev/prod topology diverges. | |
| Two Dockerfiles (Dockerfile + Dockerfile.dev) | Explicit separation; more to maintain. | |

**User's choice:** Multi-stage + override for dev HMR

### Q4 — Terminal service exec access

| Option | Description | Selected |
|--------|-------------|----------|
| Terminal also bind-mounts /var/run/docker.sock | Same socket pattern as app. | ✓ |
| Terminal goes through app's API | Significant refactor; extra hop per keystroke. | |
| Share socket-proxy if picked | Only meaningful with socket-proxy in Q2. | |

**User's choice:** Bind-mount socket directly in terminal too

## Host Paths & Preview Network

### Q1 — Reconciling containerized app with host-path bind mounts

| Option | Description | Selected |
|--------|-------------|----------|
| Mirror-mount: identical paths inside and outside | Single env var works for app code AND generated compose specs. | ✓ |
| Explicit dual vars (_HOST + _CONTAINER) | Two env vars per directory; mismatches easy to introduce. | |
| Compose project dir as named volume | Breaks 'ls on host' ergonomics; doesn't solve CLAUDE_CONFIG_PATH or PRODUCTION_APPS_DIR anyway. | |

**User's choice:** Mirror-mount

### Q2 — devdock-proxy network attachment to per-project stacks

| Option | Description | Selected |
|--------|-------------|----------|
| Update base-compose.yml template: dev container joins both networks | Template + compose-generator update; declarative. | ✓ |
| Dynamic `docker network connect` after compose up | Template untouched; imperative step with own failure modes. | |
| Caddy joins per-project networks dynamically | Inverts isolation story; reload per env. | |

**User's choice:** Update template + compose-generator

### Q3 — Caddy preview routing updates on env lifecycle

| Option | Description | Selected |
|--------|-------------|----------|
| Caddy Admin API + DevDock writes routes | POST/DELETE upstream configs; no reloads. | ✓ |
| Generate Caddyfile + reload | File I/O churn; reload race windows. | |
| Wildcard catch-all at request time | Zero runtime config; implicit routing logic via DNS + naming. | |

**User's choice:** Caddy Admin API

### Q4 — CLAUDE_CONFIG_PATH with containerized app

| Option | Description | Selected |
|--------|-------------|----------|
| Host path, mirror-mounted read-only into app | App validates existence at startup; single semantic. | ✓ |
| App doesn't touch it; pass-through to generated compose only | No mount into app; can't log-warn on misconfig. | |

**User's choice:** Mirror-mount read-only

## Startup, Dev Workflow, Scope

| Option | Description | Selected |
