# Phase 999.2 — DevDock Self-Containerization: VPS Cutover Runbook

**Status:** Ready for execution by the user on the VPS when prepared.
**Scope:** This runbook is the ONLY execution step for cutting over production from the systemd + host-Postgres + host-nginx-direct setup to the new `docker compose up` stack.

This runbook is designed to be rollback-friendly: every destructive step has an inverse, and the old systemd unit + host Postgres stay present until the user explicitly removes them in a follow-up cleanup phase.

---

## Prerequisites (verify BEFORE starting)

- [ ] DevDock repo is checked out at `/home/murx-dev/devdock` (or adjust paths below)
- [ ] Phase 999.2 Plans 01-08 + Plan 10 smoke have been executed and the current `main` branch builds locally (`docker compose build`)
- [ ] `docker` and `docker compose` v2 are installed on the VPS (`docker compose version` prints a v2.x)
- [ ] `mohed_abbas` (or the service user) is in the `docker` group: `groups mohed_abbas | grep docker`
- [ ] Host Postgres for production apps keeps running (we are NOT touching it)
- [ ] Host nginx is the TLS edge; certbot keeps managing certificates under `/etc/letsencrypt/live/...`
- [ ] Ports:
  - [ ] `127.0.0.1:8080` is free on the VPS: `ss -tlnp | grep 8080` should print nothing
  - [ ] Port 2019 is NOT intended to be exposed (confirm `docker-compose.yml` has no `2019:2019` mapping)

## Step 1 — Prepare host directories (prevent root-owned bind-mount issues)

DevDock's compose mirror-mounts host paths (D-09). If those directories don't exist at `docker compose up` time, Docker creates them as root, breaking the non-root nextjs user's writes (RESEARCH.md Pitfall 10).

```bash
sudo mkdir -p /srv/devdock/data /srv/devdock/apps-empty /srv/devdock/claude-empty
# If you use production apps monitoring:
# sudo mkdir -p /home/murx/apps  (already exists in prod per project history)
# If you use Claude Code shared config:
# Confirm your CLAUDE_CONFIG_PATH exists on host; must match the value you set in .env.local.

# Ownership: DevDock's app + terminal containers run as uid 1001 (nextjs / termuser).
# For DEVDOCK_DATA_DIR the container needs write access. Two options:
#   A) chown to uid 1001 (preferred):
sudo chown -R 1001:1001 /srv/devdock/data
#   B) Or chmod 0775 with the docker group: situational — ask an admin.
```

## Step 2 — Prepare `.env.local`

```bash
cd /home/murx-dev/devdock
cp -n .env.example .env.local   # copy only if missing; don't clobber existing secrets

# Generate the admin password hash LOCALLY (never on a shared host):
# On your laptop, NOT the VPS:
#   git clone ... && cd devdock && npm install && npm run hash-password -- 'your-strong-password'
# Copy the resulting hash into .env.local's ADMIN_PASSWORD_HASH line.

# Populate .env.local with:
#   POSTGRES_PASSWORD=<strong random>
#   AUTH_SECRET=<openssl rand -base64 32>
#   ADMIN_USERNAME=admin
#   ADMIN_PASSWORD_HASH=$2b$12$...        (pasted from local machine)
#   DEVDOCK_DATA_DIR=/srv/devdock/data
#   PRODUCTION_APPS_DIR=/home/murx         (or your actual apps dir)
#   CLAUDE_CONFIG_PATH=/home/mohed_abbas/.claude   (optional)
#   PREVIEW_DOMAIN=preview.devdock.yourdomain.com
#   NEXT_PUBLIC_PREVIEW_DOMAIN=preview.devdock.yourdomain.com
#   GITHUB_CLIENT_ID=... GITHUB_CLIENT_SECRET=... GITHUB_TOKEN_ENCRYPTION_KEY=...
#   DATABASE_URL=postgresql://devdock:<POSTGRES_PASSWORD>@postgres:5432/devdock
#   AUTH_URL=https://devdock.yourdomain.com

# Verify file perms — .env.local must NOT be world-readable:
chmod 0600 .env.local
```

## Step 3 — Build the images (dry run, no service disruption)

```bash
cd /home/murx-dev/devdock
# Use -f docker-compose.yml ONLY to skip the docker-compose.override.yml dev layer:
docker compose -f docker-compose.yml build
```

Expected: `app`, `terminal` images built successfully. ~3-5 minutes on first build.

## Step 4 — Stop systemd DevDock + host nginx reload prep

**Note:** From this point the old DevDock dashboard is offline until Step 7 finishes. Plan a short maintenance window.

```bash
sudo systemctl stop devdock
sudo systemctl disable devdock    # optional: keep enabled if you want rollback easy
sudo systemctl status devdock     # verify: inactive (dead)
```

## Step 5 — Update host nginx to proxy to Caddy :8080

The updated `deploy/nginx/devdock.conf` and `deploy/nginx/devdock-preview.conf` already point at `127.0.0.1:8080`. If your install uses those files via symlink, reloading nginx after the service-user pulls the latest repo is sufficient. Otherwise copy them into place:

```bash
# If installed via symlink (recommended):
ls -l /etc/nginx/sites-enabled/devdock.conf   # verify it points into the repo

# Verify nginx sees the updated config:
sudo nginx -t

# Do NOT reload yet — wait until Step 7 so traffic doesn't hit a dead :8080.
```

## Step 6 — Bring the stack up

```bash
cd /home/murx-dev/devdock
docker compose -f docker-compose.yml up -d --wait
```

Expected: command completes with all four services reporting healthy. Flags:
- `--wait` blocks until every service's healthcheck passes or a timeout (typically ~60s)
- On failure, inspect: `docker compose logs app`, `docker compose logs postgres`, `docker compose logs caddy`, `docker compose logs terminal`

Verify all services:
```bash
docker compose ps
# Expect 4 rows, all STATUS "Up" and healthy.

# Probe app health from inside the stack:
docker compose exec app wget -qO- http://localhost:3000/api/health

# Probe Caddy admin API (compose-internal only — never expose this port to host):
docker compose exec app wget -qO- http://caddy:2019/config/ | head -c 100
```

## Step 7 — Reload host nginx and smoke test externally

```bash
sudo nginx -t && sudo systemctl reload nginx

# From your laptop:
curl -I https://devdock.yourdomain.com/api/health
# Expect: HTTP/2 200

# Log in at https://devdock.yourdomain.com with ADMIN_USERNAME / the plaintext password
# that produced ADMIN_PASSWORD_HASH.

# Create a test environment, start it, hit the preview URL. Verify Caddy
# registered the route:
docker compose exec app wget -qO- http://caddy:2019/config/apps/http/servers/srv0/routes | head -c 500
# Expect to see a route object with @id "preview-<your-env-slug>".
```

## Step 8 — Monitor + confirm

# During the window, exercise:

## Rollback (if needed)

```bash
cd /home/murx-dev/devdock
docker compose down                  # does NOT delete volumes; migrations stay
sudo systemctl start devdock         # old Next.js systemd back online (if still enabled)
# Revert host nginx upstream to :3000:
# (checkout the previous commit of deploy/nginx/*.conf, sudo nginx -t && sudo systemctl reload nginx)
```
