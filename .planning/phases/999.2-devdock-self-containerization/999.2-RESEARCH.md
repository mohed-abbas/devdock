## Summary

This phase containerizes DevDock itself using Docker Compose: Next.js `app`, Socket.IO `terminal`, `postgres`, and `caddy` as four services in a single stack. The current runtime — systemd + host Postgres + host nginx direct to Next.js — is replaced by `docker compose up -d --build`. This research covers the ten high-research-value implementation areas locked by the 18 CONTEXT decisions.

The primary technical risks are: (1) drizzle-kit is a devDependency that must run in the production entrypoint — the safest resolution is moving it to `dependencies` and bundling it explicitly via `outputFileTracingIncludes`; (2) bcrypt requires glibc so the runner base image must be `node:22-slim` (Debian), not Alpine; (3) the `devdock-proxy` external network must be owned by DevDock's main `docker-compose.yml` with `external: false` and an explicit `name: devdock-proxy`, while per-project templates reference it with `external: true name: devdock-proxy`; (4) Caddy admin API defaults to `localhost:2019` which is unreachable from the `app` container — it must be configured to bind on `0.0.0.0:2019` (or the compose service name) within the compose-internal network, never published to the host.

**Primary recommendation:** Implement in this order: Dockerfiles → Compose files → entrypoint scripts → Caddy Admin API module → env extension → compose-generator update → HUMAN-UAT runbook. The entrypoint ordering (postgres healthcheck → drizzle-kit push → admin seed → next start) is the highest-risk code path and must be explicitly tested.

## Domain Landscape

### Next.js Standalone Output (Already Configured)

`next.config.ts` already sets `output: 'standalone'` and `outputFileTracingRoot: path.join(__dirname)`. [VERIFIED: codebase read]

When built, Next.js uses `@vercel/nft` to trace all imports and copies only needed files to `.next/standalone/`. The standalone directory can be deployed without installing `node_modules` — it includes a bundled minimal `node_modules/` and `server.js`. [CITED: https://nextjs.org/docs/app/api-reference/config/next-config-js/output]

**Critical caveat:** `public/` and `.next/static/` are NOT copied automatically to the standalone directory. The Dockerfile must explicitly copy them:
```
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
```
[CITED: https://nextjs.org/docs/app/api-reference/config/next-config-js/output]

### drizzle-kit in Production Containers

drizzle-kit is a `devDependency` in `package.json`. [VERIFIED: codebase read — `"drizzle-kit": "^0.31.10"` in devDependencies]

The `@vercel/nft` file tracer used by Next.js standalone output traces only what the app *imports at runtime* — it does not trace CLI tools invoked in entrypoints. drizzle-kit will NOT be included in `.next/standalone/node_modules/` by default. [CITED: https://nextjs.org/docs/app/api-reference/config/next-config-js/output]

**Three viable options (recommendation in Open Questions #1):**
1. Move drizzle-kit to `dependencies` (simplest; resolves tracing automatically)
2. Use `outputFileTracingIncludes` to force-include drizzle-kit binaries
3. Run migrations from a separate builder-stage binary before the runner starts (init container pattern — more complex)

drizzle-kit can read `.ts` schema files when invoked via `npx drizzle-kit push` because it bundles its own TypeScript execution (esbuild-based). No tsx/ts-node needed at entrypoint. [CITED: https://orm.drizzle.team/docs/drizzle-kit-push]

drizzle-kit push is non-interactive when there are no data-loss changes. For a first-boot scenario (clean schema push), it runs fully non-interactive. [CITED: https://orm.drizzle.team/docs/drizzle-kit-push]

**Non-interactive CI flag status:** A `--force` flag exists but has a critical limitation — it does NOT suppress all prompts in all versions; it specifically auto-accepts data-truncation statements. A `--safe` flag PR (#4384) was open as of April 2025 and not yet merged. [CITED: https://github.com/drizzle-team/drizzle-orm/pull/4384] [CITED: https://github.com/drizzle-team/drizzle-orm/issues/4921]

**Practical strategy for this phase:** On a fresh Postgres container (empty DB), `drizzle-kit push` with the existing `drizzle.config.ts` runs fully non-interactive — no prompts for CREATE TABLE/CREATE TYPE operations. The risky prompt scenario (data loss on schema change) only occurs when the schema changes after first boot. For the me-first use case, `drizzle-kit push` without `--force` is safe for fresh starts; pair with `--force` to handle the constraint-addition edge case.

### Caddy Admin API

Default admin listen address is `localhost:2019`. In a container, `localhost` is the container itself — the `app` container cannot reach `caddy:2019` by default. [CITED: https://caddyserver.com/docs/api]

**Fix:** Configure Caddy to bind admin on `0.0.0.0:2019`. Caddyfile global syntax:
```
{
    admin :2019
}
```
JSON config equivalent:
```json
{ "admin": { "listen": "0.0.0.0:2019" } }
```
[CITED: https://caddyserver.com/docs/caddyfile/options] [CITED: https://caddy.community/t/question-on-admin-api-from-a-docker-container/8724]

When binding to a wildcard interface (`0.0.0.0`), Caddy disables its Host-header validation, meaning any container on the compose-internal network can reach the admin API. Never publish port 2019 to the host. [CITED: https://caddy.community/t/question-on-admin-api-from-a-docker-container/8724]

**Admin API HTTP semantics:** [CITED: https://caddyserver.com/docs/api]
- `POST /config/[path]` — Sets or replaces object; **appends to array** if path points to an array
- `PUT /config/[path]` — Creates new object; inserts into array
- `PATCH /config/[path]` — Replaces existing object or array element
- `DELETE /config/[path]` — Removes configuration at path
- `POST /load` — Replaces entire active configuration (zero-downtime)

**@id tag:** Embed `"@id": "some-unique-id"` in any JSON object. Then access/delete via `/id/some-unique-id` without computing the array index. Ideal for routes that are added/removed dynamically. [CITED: https://caddyserver.com/docs/api]

**Route JSON shape for preview routing:**
```json
{
  "@id": "preview-{env-slug}",
  "match": [{ "host": ["{env-id}.preview.devdock.example.com"] }],
  "handle": [{
    "handler": "reverse_proxy",
    "upstreams": [{ "dial": "devdock-{slug}-dev:{preview_port}" }]
  }],
  "terminal": true
}
```

**POST to add route:** `POST http://caddy:2019/config/apps/http/servers/srv0/routes`

**DELETE route by @id:** `DELETE http://caddy:2019/id/preview-{env-slug}`

**Idempotency:** The Admin API has no native upsert. On duplicate POST to a routes array, Caddy appends (creates a second route). The application code must check existence (GET by @id → 404 means absent) or use a "delete-then-add" pattern for environment restart events. [CITED: https://caddyserver.com/docs/api — "POST appends to array"]

### bcrypt Native Module and Alpine Incompatibility

`bcrypt` version `^6.0.0` is in `dependencies`. It is a native Node.js addon requiring pre-built binaries linked against glibc. [VERIFIED: package.json]

Alpine Linux uses musl libc. Most bcrypt prebuilt binaries target glibc. On Alpine, `npm install bcrypt` triggers node-gyp compilation which requires gcc, make, python3, libstdc++ — making builds slower and potentially failing. [CITED: multiple sources, search result 2026-04-18]

**Conclusion:** Runner base image must be `node:22-slim` (Debian Bookworm slim), not `node:22-alpine`. This is a factual compatibility constraint, not a preference. [CITED: https://medium.com/@subhampradhan966/%EF%B8%8F-slim-vs-alpine-key-differences-b23ed1e693df]

## Implementation Patterns

### 1. Multi-Stage Dockerfile for `app` (D-08)

Three stages: `deps` → `builder` → `runner`.

```dockerfile
# Stage 1: deps — install all dependencies (dev + prod) for build
FROM node:22-slim AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci

# Stage 2: builder — build Next.js standalone
FROM node:22-slim AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

# Stage 3: runner — minimal runtime image
FROM node:22-slim AS runner
WORKDIR /app
ENV NODE_ENV=production

# Create non-root user matching DOCKER_GID at runtime (see entrypoint pattern below)
RUN groupadd --gid 1001 nodejs && \
    useradd --uid 1001 --gid nodejs --shell /bin/bash --create-home nextjs

# Copy standalone output
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

# Copy drizzle config + schema for entrypoint migrations
# drizzle-kit must be available in runner — move to dependencies OR copy explicitly:
COPY --from=deps --chown=nextjs:nodejs /app/node_modules/.bin/drizzle-kit ./node_modules/.bin/
COPY --from=deps --chown=nextjs:nodejs /app/node_modules/drizzle-kit ./node_modules/drizzle-kit
COPY --from=builder --chown=nextjs:nodejs /app/drizzle.config.ts ./drizzle.config.ts
COPY --from=builder --chown=nextjs:nodejs /app/src/lib/db/schema.ts ./src/lib/db/schema.ts

COPY --chown=nextjs:nodejs docker/app-entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

# GID fixup + entrypoint run as root initially, drops to nextjs after group adjustment
ENTRYPOINT ["/entrypoint.sh"]
CMD ["node", "server.js"]
```

**Note on drizzle-kit bundling:** The cleanest path is moving `drizzle-kit` from `devDependencies` to `dependencies` in `package.json`. This ensures `npm ci` in the `deps` stage installs it, and the `COPY --from=deps` pattern is then straightforward. The runner image gains ~15MB but eliminates a class of "migration runner not found" startup failures. [RECOMMENDATION: see Open Questions #1]

**Note on tsx for terminal:** `server/terminal-server.ts` currently runs via `tsx`. Recommend compiling it with `tsc` in the terminal Dockerfile (or a second stage that runs `npx tsx --tsconfig server/tsconfig.json server/terminal-server.ts` from the builder-stage deps). For me-first simplicity: install `tsx` in the terminal container's dependencies and run `node -r tsx/cjs server/terminal-server.ts` in production — avoids a separate build stage.

### 2. `app` Entrypoint Script: Startup Ordering (D-15)

```bash
#!/bin/bash
set -e

# 1. Wait for Postgres to be reachable (relies on depends_on: service_healthy in compose)
echo "Waiting for postgres..."
until pg_isready -h postgres -U "${POSTGRES_USER:-devdock}" -d "${POSTGRES_DB:-devdock}"; do
  sleep 1
done
echo "Postgres ready."

# 2. Runtime docker GID fix (see GID pattern below)
if [ -S /var/run/docker.sock ]; then
  DOCKER_SOCK_GID=$(stat -c '%g' /var/run/docker.sock)
  if ! getent group "${DOCKER_SOCK_GID}" > /dev/null 2>&1; then
    groupadd --gid "${DOCKER_SOCK_GID}" docker-host
  fi
  usermod -aG "$(getent group ${DOCKER_SOCK_GID} | cut -d: -f1)" nextjs
fi

# 3. Run migrations (drizzle-kit push)
echo "Running migrations..."
npx drizzle-kit push --config=drizzle.config.ts
echo "Migrations complete."

# 4. Admin seed (idempotent — guards on empty users table)
echo "Checking admin seed..."
npx tsx src/scripts/seed-admin-boot.ts
echo "Seed complete."

# 5. Start server (exec preserves PID 1 for signal handling)
exec "$@"
```

`pg_isready` requires the `postgresql-client` package in the runner image. Add to Dockerfile:
```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends postgresql-client && rm -rf /var/lib/apt/lists/*
```

### 3. Docker Socket GID Mapping (D-06, D-07) — Runtime Detection Pattern

**Problem:** On Linux, `/var/run/docker.sock` is owned by the `docker` group (typically GID 999 on Debian, but varies). Building the GID into the image via a `--build-arg` bakes in a host-specific value that breaks portability (D-16 requires same compose on macOS dev and Linux prod).

**Solution:** Runtime detection in the entrypoint (shown above). The pattern:
```bash
DOCKER_SOCK_GID=$(stat -c '%g' /var/run/docker.sock)
if ! getent group "${DOCKER_SOCK_GID}" > /dev/null 2>&1; then
  groupadd --gid "${DOCKER_SOCK_GID}" docker-host
fi
usermod -aG "$(getent group ${DOCKER_SOCK_GID} | cut -d: -f1)" nextjs
```

**macOS caveat:** On Docker Desktop for Mac, the socket is a proxy — socket GID is irrelevant (any user can read the mounted socket). The runtime detection still runs but `stat -c '%g'` on the Mac socket may return 0 (root). This is harmless: `usermod -aG root nextjs` adds the user to root group, but since the socket is world-accessible on Mac it doesn't matter. [ASSUMED — macOS Docker Desktop socket behavior from training knowledge; verify on dev machine if issues arise]

**The entrypoint must run as root initially** to execute `groupadd/usermod`, then switch to `nextjs` user with `exec su-exec nextjs "$@"` or by using `gosu`. Alternative: use `su-exec` (Alpine) or `gosu` (Debian) — install in Dockerfile, use at end of entrypoint.

### 5. `devdock-proxy` External Network (D-10)

**Pattern (verified):** The main DevDock `docker-compose.yml` defines the network with `external: false` (or omitted) and an explicit `name:`. Per-project composes reference it with `external: true` plus matching `name:`. [CITED: http://blog.differentpla.net/blog/2025/03/30/docker-compose-shared-networks/] [CITED: https://docs.docker.com/reference/compose-file/networks/]

**In DevDock's `docker-compose.yml`:**
```yaml
networks:
  devdock-proxy:
    name: devdock-proxy
    driver: bridge
    ipam:
      config:
        - subnet: 172.30.0.0/24
```

**In each per-project `docker-compose.yml` (generated template):**
```yaml
networks:
  project-net:
    name: "devdock-{{PROJECT_SLUG}}-net"
    driver: bridge
  devdock-proxy:
    name: devdock-proxy
    external: true

services:
  dev:
    networks:
      - project-net
      - devdock-proxy
```

**compose-generator.ts change:** Add the `devdock-proxy` network declaration and attach the `dev` service to it. The `name: devdock-proxy` in the external block is required for Docker to locate the correct network across compose projects. [CITED: https://docs.docker.com/reference/compose-file/networks/]

**Network creation ordering:** Docker Compose `up` creates the network when it doesn't exist. If DevDock's stack is `up` and a user project tries to attach to `devdock-proxy` as external before DevDock is up, Compose returns an error. This is acceptable — user environments cannot start without DevDock running.

**Subnet collision avoidance (Claude's discretion):** Use `172.30.0.0/24` for `devdock-proxy`. Per-project networks default to Docker's automatic subnet allocation (typically `172.17.x.x`–`172.27.x.x`). Using `172.30.x.x` range for the shared proxy network avoids collisions with Docker's default pool. [ASSUMED — subnet range availability from training knowledge; verify `docker network ls` on VPS before deployment]

**[BLOCKING] Caddy service must attach to BOTH networks:** The `caddy` service in DevDock's main `docker-compose.yml` must be connected to BOTH `devdock-net` (compose-internal — can reach `app:3000` and `terminal:3001`) AND `devdock-proxy` (shared — can reach `devdock-{slug}-dev` containers for preview upstreams). Without `devdock-proxy`, Caddy cannot resolve per-project dev container names and preview routing fails with upstream connection errors.

```yaml
# DevDock's docker-compose.yml — caddy service network attachment
caddy:
  networks:
    - devdock-net    # reaches app + terminal
    - devdock-proxy  # reaches per-project dev containers for preview proxy
```

This is the network topology requirement that makes preview routing work end-to-end: host nginx → Caddy (devdock-net, devdock-proxy) → `devdock-{slug}-dev:{preview_port}` (devdock-proxy).

### 6. Caddy Admin API Module Pattern (D-11)

New file: `src/lib/docker/caddy-client.ts`

```typescript
const CADDY_ADMIN_URL = process.env.CADDY_ADMIN_URL || 'http://caddy:2019';

export async function addPreviewRoute(
  envSlug: string,
  envId: string,
  previewDomain: string,
  containerName: string,
  previewPort: number,
): Promise<void> {
  const routeId = `preview-${envSlug}`;
  const hostname = `${envId}.${previewDomain}`;

  const route = {
    '@id': routeId,
    match: [{ host: [hostname] }],
    handle: [{
      handler: 'reverse_proxy',
      upstreams: [{ dial: `${containerName}:${previewPort}` }],
    }],
    terminal: true,
  };

  // Delete existing route by @id first (idempotent upsert pattern)
  await fetch(`${CADDY_ADMIN_URL}/id/${routeId}`, { method: 'DELETE' }).catch(() => {});

  // POST to routes array (appends)
  const res = await fetch(
    `${CADDY_ADMIN_URL}/config/apps/http/servers/srv0/routes`,
    {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(route),
    }
  );
  if (!res.ok) {
    throw new Error(`Caddy admin API error: ${res.status} ${await res.text()}`);
  }
}

export async function removePreviewRoute(envSlug: string): Promise<void> {
  const routeId = `preview-${envSlug}`;
  const res = await fetch(`${CADDY_ADMIN_URL}/id/${routeId}`, { method: 'DELETE' });
  if (!res.ok && res.status !== 404) {
    throw new Error(`Caddy admin API error: ${res.status}`);
  }
}
```

**Server name `srv0`:** The server name in the Caddy JSON config determines the path. Caddyfile-generated configs use `srv0` by default. Verify by calling `GET http://caddy:2019/config/apps/http/servers/` after Caddy starts to confirm the server key. [ASSUMED — "srv0" is the standard Caddyfile-to-JSON name; verify via GET on admin API on first boot]

### 7. Admin Seed from Environment Variables (D-14)

New file: `src/scripts/seed-admin-boot.ts` (non-interactive, separate from interactive `seed-admin.ts`).

Logic:
1. Read `ADMIN_USERNAME` and `ADMIN_PASSWORD_HASH` from env
2. If either is missing, log warning and exit 0 (don't block startup)
3. Query `SELECT count(*) FROM users` — if > 0, exit 0 (already seeded)
4. Insert user with role='admin' using the pre-hashed password
5. Exit 0 on success

```typescript
import { drizzle } from 'drizzle-orm/node-postgres';
import { Pool } from 'pg';
import { users } from '../lib/db/schema';
import { count } from 'drizzle-orm';

async function seedAdmin() {
  const username = process.env.ADMIN_USERNAME;
  const passwordHash = process.env.ADMIN_PASSWORD_HASH;

  if (!username || !passwordHash) {
    console.log('Seed: ADMIN_USERNAME or ADMIN_PASSWORD_HASH not set — skipping seed');
    return;
  }

  const pool = new Pool({ connectionString: process.env.DATABASE_URL! });
  const db = drizzle({ client: pool });

  try {
    const [{ value }] = await db.select({ value: count() }).from(users);
    if (value > 0) {
      console.log('Seed: users table not empty — skipping seed');
      return;
    }

    await db.insert(users).values({ username, passwordHash, role: 'admin' });
    console.log(`Seed: admin user "${username}" created`);
  } finally {
    await pool.end();
  }
}

seedAdmin().catch((err) => {
  console.error('Seed error:', err.message);
  process.exit(1);
});
```

**hash-password helper:** Create `src/scripts/hash-password.ts` (invoked via tsx, not an inline node -e one-liner — the node -e pattern is fragile due to quoting and argv indexing differences across shells):

```typescript

if (!password) { console.error('Usage: npx tsx src/scripts/hash-password.ts <password>'); process.exit(1); }

```


This invokes bcrypt with rounds=12 (matching `seed-admin.ts`'s `SALT_ROUNDS = 12`). [VERIFIED: codebase — seed-admin.ts uses `SALT_ROUNDS = 12`]

## Pitfalls & Landmines

### Pitfall 1: drizzle-kit Not Found in Runner Image
**What goes wrong:** `drizzle-kit` is a devDependency. Next.js standalone output excludes CLI tools not imported at runtime. Entrypoint runs `npx drizzle-kit push` → `command not found`.

**Why it happens:** `@vercel/nft` traces import graphs, not shell commands.

**How to avoid:** Move `drizzle-kit` to `dependencies` (simplest) OR explicitly copy `node_modules/drizzle-kit` and its esbuild/jiti deps in the Dockerfile with `COPY --from=deps`.

**Warning signs:** Container starts, runs entrypoint, logs `sh: drizzle-kit: not found` and exits.

### Pitfall 2: drizzle-kit push Hangs on Data-Loss Prompt
**What goes wrong:** Schema has a column removal or type change since last push. drizzle-kit shows an interactive prompt for confirmation. Container startup hangs indefinitely with no TTY.

**Why it happens:** `--force` does not suppress all prompts in all drizzle-kit versions; the PR (#4384) for a `--safe` flag was open as of April 2025.

**How to avoid:** For a fresh Postgres volume (first boot, blank DB), there are no existing tables to prompt about. CREATE TABLE operations are always non-interactive. The risk is on subsequent deployments where schema changes require data migration. Accept `--force` for now (me-first phase); plan to switch to `drizzle-kit migrate` in OSS distribution phase per CONTEXT deferred items.

**Warning signs:** `docker logs app` shows drizzle-kit output but no "Migrations complete" line and container doesn't start.

### Pitfall 3: Caddy Admin API Unreachable (Default localhost:2019)
**What goes wrong:** `caddy-client.ts` makes HTTP requests to `http://caddy:2019`. Caddy is running but admin API is on `localhost:2019` inside the Caddy container (unreachable from `app`).

**Why it happens:** Default Caddy admin bind is `localhost:2019` — container-local only.

**How to avoid:** Set `admin :2019` in the Caddyfile global block. NEVER publish port 2019 to the host (only expose on compose-internal network).

**Warning signs:** `caddy-client.ts` throws `ECONNREFUSED` on `http://caddy:2019`.

### Pitfall 4: Caddy Admin Route Array Grows Unbounded on Env Restarts
**What goes wrong:** Environment is stopped and started multiple times. Each time `addPreviewRoute` is called, it POSTs to the routes array (appending). Same @id tag appears multiple times. Eventually the first matching route wins, but the config grows.

**Why it happens:** Caddy Admin API POST to array appends; no upsert semantics.

**How to avoid:** Use the delete-then-add pattern: `DELETE /id/preview-{slug}` (ignoring 404) before every `POST`. This is idempotent.

**Warning signs:** `GET http://caddy:2019/config/apps/http/servers/srv0/routes` returns duplicate entries with same @id.

### Pitfall 5: Docker Socket GID Mismatch — Permission Denied
**What goes wrong:** App container runs as `nextjs` user. `/var/run/docker.sock` GID on VPS is 999 (docker group). Container's `nextjs` user is not in a group with GID 999. `dockerode` calls fail with `EACCES`.

**Why it happens:** Socket GID baked at build time doesn't match host; or runtime detection was skipped.

**How to avoid:** Runtime GID detection in entrypoint (`stat -c '%g' /var/run/docker.sock`). Entrypoint must run as root initially, perform group fix, then drop to `nextjs`. Use `gosu` for clean privilege drop.

**Warning signs:** `docker-service.ts` operations fail with `Error: connect EACCES /var/run/docker.sock`.

### Pitfall 6: bcrypt Compilation Failure on Alpine
**What goes wrong:** Base image changed to `node:22-alpine` for smaller size. `npm ci` triggers `node-gyp` for bcrypt. Build fails or is very slow due to missing gcc/musl headers.

**Why it happens:** bcrypt native addon ships glibc prebuilts; Alpine uses musl.

**How to avoid:** Always use `node:22-slim` (Debian) for any stage that installs bcrypt. Never use Alpine.

**Warning signs:** Build stage output shows `gyp` compilation errors or `make` not found.

### Pitfall 7: NEXT_PUBLIC_PREVIEW_DOMAIN Baked at Wrong Stage
**What goes wrong:** `NEXT_PUBLIC_PREVIEW_DOMAIN` is a build-time env var (inlined into JS bundle by Next.js). It's set in `.env.local` on the host but not passed as a Docker build arg. Preview buttons show undefined or the dev domain value in production.

**Why it happens:** `NEXT_PUBLIC_*` vars are resolved during `next build`, not at runtime.

**How to avoid:** Pass as `ARG`/`ENV` in Dockerfile builder stage, or use `--build-arg` in compose's `build.args`. Add to `docker-compose.yml`:
```yaml
app:
  build:
    args:
      - NEXT_PUBLIC_PREVIEW_DOMAIN=${NEXT_PUBLIC_PREVIEW_DOMAIN}
```

**Warning signs:** Preview button in dashboard shows undefined domain or "localhost:3000".

### Pitfall 8: Caddy Server Key Is Not "srv0"
**What goes wrong:** `caddy-client.ts` posts routes to `/config/apps/http/servers/srv0/routes`. Caddy uses a different server key (e.g., `main`, `devdock`, etc.). Routes are added to a non-existent path silently, or return 400.

**Why it happens:** The server key depends on how the Caddyfile is structured. `srv0` is the default when using Caddyfile with a single `:port` block, but it's not guaranteed.

**How to avoid:** In the Caddy Admin API module's initialization, call `GET http://caddy:2019/config/apps/http/servers/` and discover the actual server key at startup. Or set an explicit `@id` on the server block itself.

**Warning signs:** `addPreviewRoute` succeeds (HTTP 200) but no route appears when checking `/config/apps/http/servers/`.

### Pitfall 9: devdock-proxy Network Missing When Per-Project Compose Runs
**What goes wrong:** User creates an environment before DevDock's compose stack is fully up. Per-project compose references `devdock-proxy` as `external: true`. Docker Compose returns error: "network devdock-proxy declared as external, but could not be found."

**Why it happens:** DevDock stack creates the network when it starts up. Race condition or DevDock not running.

**How to avoid:** This is expected behavior — user environments can't start without DevDock running. Document in the runbook: "DevDock must be running before environments can be created." The app should catch `composeUp` errors and surface a clear message.

**Warning signs:** `docker-service.ts` `composeUp` returns error containing "network devdock-proxy ... could not be found".

### Pitfall 10: Mirror-Mount Path Used as Bind Source Before Container Init
**What goes wrong:** `DEVDOCK_DATA_DIR=/srv/devdock/data` set in `.env.local`. Container starts. Entrypoint creates directories. But the host path doesn't exist yet — Docker silently creates it as root-owned, breaking permissions.

**Why it happens:** Docker bind-mount creates the host directory as root if it doesn't exist.

**How to avoid:** Pre-create host directories in the VPS cutover runbook before `docker compose up`. Document in HUMAN-UAT.md: `mkdir -p /srv/devdock/data`.

**Warning signs:** `EACCES` errors when writing to data directories after first boot.

## Security Threats & Mitigations

### Threat 1: Docker Socket Confused-Deputy Attack [BLOCKING]

**Threat:** The `app` and `terminal` containers have `/var/run/docker.sock` mounted. A vulnerability in DevDock's web layer (SSRF, RCE, injection) could allow an attacker to issue arbitrary Docker commands via the socket — creating privileged containers, reading host files, or lateral movement to production containers.

**STRIDE:** Elevation of Privilege, Information Disclosure

**Mitigations:**
- Docker socket ONLY in `app` + `terminal` services (INFRA-04). Never in `caddy`, `postgres`, or user-project containers. [VERIFIED: CONTEXT D-06/D-07]
- `app` and `terminal` containers run as non-root user `nextjs`/`node` — socket access via supplemental group only
- Per-project dev containers have no socket mount (verified: `base-compose.yml` contains explicit `# Docker socket is NEVER mounted here (INFRA-04)` comment)
- Input to `execFile` (docker compose CLI) is array-form — no shell injection possible. [VERIFIED: docker-service.ts uses `execFile` not `exec`]

### Threat 2: Caddy Admin API Internet Exposure [BLOCKING]

**Threat:** Caddy admin API binds to `0.0.0.0:2019` within the container. If port 2019 is published to the host (e.g., accidentally added to compose `ports:`), it becomes internet-accessible. Anyone can rewrite Caddy's routing table, redirect all traffic to an attacker-controlled server.

**STRIDE:** Tampering, Denial of Service

**Mitigations:**
- Port 2019 MUST NOT appear in compose `ports:`. Admin API is on compose-internal network only.
- Caddy container's only published port should be the internal port (e.g., `127.0.0.1:8080:8080`)
- Add explicit comment in `docker-compose.yml`: `# DO NOT publish port 2019 to host — admin API internal only`
- Plan-checker should flag any `2019:2019` in ports section

### Threat 3: Admin Seed Credential Leakage [BLOCKING]

**Threat:** `ADMIN_PASSWORD_HASH` stored as plaintext bcrypt hash in `.env.local`. If `.env.local` is committed to git (accidental), or if `docker inspect` shows env vars, the hash is exposed. An attacker with the hash can attempt offline cracking (bcrypt rounds=12 slows but doesn't prevent this).

**STRIDE:** Information Disclosure

**Mitigations:**
- `.env.local` is gitignored (existing project convention). [VERIFIED: .env.example exists; .env.local is implied gitignored per CONTEXT]
- Store hash, never plaintext. bcrypt rounds=12 means ~0.3s per attempt — brute force is slow.
- `ADMIN_PASSWORD_HASH` is never logged in entrypoint output
- Use `docker inspect --format='{{.Config.Env}}'` only with access already established
- Document in HUMAN-UAT.md: "Never commit .env.local; audit git history after cutover"

### Threat 4: Postgres Exposed on Compose Network

**Threat:** Postgres runs on the compose-internal network. If another compose project's containers join that network (misconfigured external network reference), they could reach DevDock's Postgres on port 5432.

**STRIDE:** Information Disclosure, Tampering

**Mitigations:**
- DevDock's compose network (`devdock-net`) is distinct from `devdock-proxy`. `devdock-proxy` is for preview routing only — Postgres is NOT attached to `devdock-proxy`.
- Postgres service should only be on the `devdock-net` internal compose network
- Use a strong, randomly generated `POSTGRES_PASSWORD` in `.env.local` (not the default)
- `POSTGRES_PASSWORD` should not appear in `DATABASE_URL` in .env.example (leave template only)

### Threat 5: Terminal Token Leakage via Headers

**Threat:** Short-lived terminal tokens are passed in Socket.IO auth. If Caddy or nginx logs the WebSocket handshake URL (which may include query parameters), tokens could appear in access logs.

**STRIDE:** Information Disclosure

**Mitigations:** Terminal auth uses `socket.handshake.auth?.token` (body, not query param for auth namespace). The logs namespace may use query param fallback. [VERIFIED: terminal-server.ts line 179 — `socket.handshake.query?.token`] Ensure Caddy and nginx access logs do not log request bodies or query strings for WebSocket upgrade requests. Caddy's default log format does not log query params. nginx default `combined` format logs the request path including query.

## Open Questions (Claude's Discretion Recommendations)

### 1. drizzle-kit bundling strategy [RECOMMENDATION: move to dependencies]

**Options:** (a) Move `drizzle-kit` to `dependencies` (simplest, +~15MB runner), (b) `outputFileTracingIncludes` to force-include, (c) init-container pattern.

**Recommendation:** Move to `dependencies`. This phase is me-first and the size is negligible. Using `outputFileTracingIncludes` requires knowing the exact file paths inside drizzle-kit's package, which changes with versions — fragile. The init-container pattern adds orchestration complexity.

### 2. DOCKER_GID: compile-time build arg vs runtime detection [RECOMMENDATION: runtime detection]

**Options:** (a) `--build-arg DOCKER_GID=999` baked at build time, (b) runtime detection via `stat -c '%g' /var/run/docker.sock` in entrypoint.

**Recommendation:** Runtime detection. Build-arg approach requires rebuilding the image when deployed to a host with a different docker group GID, and breaks the D-16 goal (same compose on macOS and Linux). macOS harmlessly detects GID 0 — no functional impact since Docker Desktop socket is world-accessible.

### 3. Caddy Admin API auth [RECOMMENDATION: internal network isolation is sufficient]

**Options:** (a) No auth — rely on compose-internal network isolation, (b) Add an admin key via Caddy's `admin.config.enforce_origin`, (c) Use unix socket for admin instead of TCP.

**Recommendation:** Internal network isolation is sufficient for this phase. The admin API is never published to the host (port 2019 not in compose `ports:`). Only the `app` container can reach it. Adding an `CADDY_ADMIN_KEY` env var adds complexity with little security gain when the network perimeter is compose-internal. Re-evaluate for multi-user phase if admin API calls originate from user-accessible code paths.

### 4. Exact internal Caddy port [RECOMMENDATION: 8080]

**Options:** Any high port not in use by production services.

**Recommendation:** `8080`. Rationale: (a) convention for HTTP alt port, (b) easy to remember, (c) unlikely to conflict with existing VPS services on 127.0.0.1. Verify with `ss -tlnp | grep 8080` on VPS before cutover.

### 5. Compose file naming: `docker-compose.yml` vs `compose.yaml` [RECOMMENDATION: `docker-compose.yml`]

**Recommendation:** `docker-compose.yml` for compatibility with older Docker Compose versions still installed on some Linux VPSes. Both names work with Docker Compose v2+, but `docker-compose.yml` has broader tooling support (CI, scripts, documentation references).

### 6. Terminal service DB access [RECOMMENDATION: not needed]

**Finding:** `server/terminal-server.ts` uses only `dockerode` (via `docker-service.ts`) and the `verifySignedToken` auth module. It does NOT import any DB client or Drizzle. [VERIFIED: codebase read]

**Recommendation:** `terminal` service does not need a database connection. No `DATABASE_URL` in the terminal container env. `depends_on: app: condition: service_started` is optional but provides a logical ordering guarantee.

### 7. Log driver and rotation [RECOMMENDATION: json-file with rotation]

```yaml
logging:
  driver: json-file
  options:
    max-size: "10m"
    max-file: "3"
```
Apply to all four services. This mirrors the daemon-level log rotation already recommended in PITFALLS.md. Per-service configuration ensures consistency regardless of Docker daemon defaults.

### 8. tsx in terminal container: runtime vs compiled [RECOMMENDATION: runtime tsx]

**Options:** (a) Run `server/terminal-server.ts` via tsx at runtime (`npx tsx server/terminal-server.ts`), (b) Compile with tsc to JS in a build stage.

**Recommendation:** Runtime tsx for me-first simplicity. `tsx` is already in devDependencies and is fast (~10ms overhead). Move `tsx` to `dependencies` alongside `drizzle-kit`. Terminal container shares the same deps stage as app. No extra build step, no `tsconfig.json` for terminal separate from app.

### 9. Postgres version pin [RECOMMENDATION: postgres:16-alpine]

**Recommendation:** `postgres:16-alpine` for the compose `postgres` service. Alpine is fine for Postgres (unlike Node.js bcrypt). Smaller image. `16` matches the production Postgres version on VPS and the schema research. Do NOT use `postgres:latest` — pin to major version.

### 10. devdock-proxy subnet [RECOMMENDATION: 172.30.0.0/24]

Recommendation: `172.30.0.0/24`. Avoids Docker's default allocation range (172.17.x.x–172.27.x.x) and per-project compose network ranges. Single /24 is sufficient for Caddy + all user-project dev containers that need preview routing. Verify no clash on VPS: `docker network ls` + `ip route show` before cutover.
