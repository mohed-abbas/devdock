## Gaps Discovered During Plan 10 E2E Validation

| Gap ID | Affected Task | Symptom | Root Cause | Fix Scope |
|--------|---------------|---------|------------|-----------|
| GAP-1 | 04-T1 / 01-T3 | `tests/entrypoint.test.sh` times out after 60s waiting for postgres | `scripts/entrypoint-app.sh` defaults `POSTGRES_HOST=postgres` and ignores `POSTGRES_PORT`; the test runs entrypoint on the host against a throwaway postgres on `127.0.0.1:55432` but never overrides those vars. Test exports `DATABASE_URL` only, which the wait loop ignores. | Patch entrypoint to honor `POSTGRES_PORT` (already used implicitly by `pg_isready -p`); patch test to export `POSTGRES_HOST=127.0.0.1` and `POSTGRES_PORT=55432`. |
| GAP-2 | 03-T1 / 10-T1 | `docker compose up --build` fails at `RUN npm run build` with `Invalid environment variables: { DATABASE_URL: [ 'Required' ], AUTH_SECRET: [ 'Required' ] }` | Next.js 15 page-data collection at build time evaluates `src/lib/config.ts` which calls `loadConfig()` → throws on missing vars. The Dockerfile builder stage provides `NEXT_PUBLIC_PREVIEW_DOMAIN` only; runtime-only vars never see placeholders. | Builder stage of Dockerfile must provide harmless build-time placeholders (e.g., `ARG DATABASE_URL=postgresql://placeholder@localhost:5432/placeholder` and `ARG AUTH_SECRET=build-time-placeholder-32chars-min`) so page-data collection succeeds. Real values come from compose `env_file` at runtime. |
| GAP-3 | 10-T1 (caddy startup) | ✅ CLOSED by 999.2.1 | Caddy container fail-loops: `Error: adapting config using caddyfile: parsing caddyfile tokens for 'persist_config': persist_config must be 'off'`. Surfaced after GAP-2 unblocked the build stage. | `Caddyfile:26` had `persist_config on`; Caddy's Caddyfile global option only accepts `off` (it exists solely to disable the default-on persistence). Original intent (dynamic preview routes survive restarts) is satisfied by Caddy's default-on behavior when the directive is omitted. | Remove the line; keep the explanatory comment. (Fixed in commit `1878cf2`.) |
| GAP-4 | 10-T1 (compose up) | ⚠️ ROUTED TO 999.2.2 | `docker compose up` fails: `mounts denied: The path /srv/devdock/data is not shared from the host`. | `docker-compose.yml` defaults `DEVDOCK_DATA_DIR`, `PRODUCTION_APPS_DIR`, `CLAUDE_CONFIG_PATH` to production VPS absolute paths. The mount spec pattern `${VAR}:${VAR}` requires absolute paths on both sides; relative dev paths break container-path validation. `.env.example` documents only `DEVDOCK_DATA_DIR=./data` (relative), which also fails. | Update compose mount pattern to `${VAR}:/fixed/container/path` so host path can be relative in dev, OR update `.env.example` to document the absolute-path requirement for local dev. |
| GAP-5 | 10-T1 (terminal boot) | ⚠️ ROUTED TO 999.2.2 | `devdock-terminal` container loops unhealthy: `Invalid environment variables: { DATABASE_URL: [ 'Required' ] }` at `loadConfig()`. | `docker-compose.yml:145-153` terminal service env block is missing `DATABASE_URL`. `server/terminal-server.ts` imports `src/lib/config.ts` which calls `loadConfig()` at module-init and hard-fails. App service (line 88) correctly sets it; terminal was overlooked. | Add `DATABASE_URL: postgresql://${POSTGRES_USER:-devdock}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB:-devdock}` to the terminal env block, OR split `src/lib/config.ts` so terminal-only paths don't require DB vars. |
