    ```yaml
    # docker-compose.override.yml — Dev HMR overlay (Phase 999.2, D-16)
    #
    # Auto-loaded by `docker compose up` (compose v2 merges this on top of docker-compose.yml).
    # To run the production stack without dev overrides, use:
    #   docker compose -f docker-compose.yml up -d --build

    services:
      app:
        # In dev, re-use the `deps` stage (has all devDependencies) instead of the minimal runner.
        build:
          target: deps
        # Override CMD to Next.js dev server with turbopack HMR.
        command: ["npx", "next", "dev", "--turbopack", "--port", "3000", "--hostname", "0.0.0.0"]
        # Bind-mount source directories so edits on the host trigger HMR inside the container.
        # node_modules intentionally NOT mounted — we use the image's copy (platform-correct).
        volumes:
          - ./src:/app/src
          - ./server:/app/server
          - ./public:/app/public
          - ./next.config.ts:/app/next.config.ts
          - ./tsconfig.json:/app/tsconfig.json
          - ./package.json:/app/package.json
          - ./drizzle.config.ts:/app/drizzle.config.ts
          - ./middleware.ts:/app/middleware.ts
          # Anonymous volume preserves image node_modules / .next from host mount bleed.
          - /app/node_modules
          - /app/.next
        # Keep the production env/mounts from docker-compose.yml via merge semantics.
        # Override NODE_ENV for dev mode:
        environment:
          NODE_ENV: development
        # Disable the entrypoint in dev — no drizzle-kit push / seed needed from app
        # since a developer has usually already done it. Run manually when needed:
        #   docker compose exec app npx drizzle-kit push
        #   docker compose exec app npm run seed-admin
        entrypoint: []

      terminal:
        build:
          target: deps
        command: ["npx", "tsx", "--watch", "server/terminal-server.ts"]
        volumes:
          - ./src:/app/src
          - ./server:/app/server
          - ./tsconfig.json:/app/tsconfig.json
          - ./package.json:/app/package.json
          - /app/node_modules
        environment:
          NODE_ENV: development
        entrypoint: []
    ```

    Notes:

    - The dev override does NOT override postgres or caddy — they work identically in dev and prod.

|----------|-------------|
| host → caddy :8080 | Loopback-only bind; trust relies on host nginx being the only caller |
| compose-internal network → postgres :5432 | Only devdock-net members can reach; no host exposure |
| caddy :2019 admin | Compose-internal only; any network member can call |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.2-01 | Elevation of Privilege | Docker socket mount scope | mitigate | Socket bind-mount present ONLY on `app` and `terminal`. `caddy` and `postgres` services have NO socket mount. Enforced by structure; acceptance criteria grep verifies. |
| T-999.2-02 | Tampering, DoS | Caddy admin port exposure | mitigate | No `2019:2019` mapping anywhere in compose. Caddy's only published port is `127.0.0.1:${CADDY_INTERNAL_PORT}:8080` — loopback bind means even if port 2019 leaked via firewall misconfig, host nginx alone can reach it. |
| T-999.2-04 | Information Disclosure | Postgres network exposure | mitigate | Postgres has no `ports:` mapping. Reachable only via `devdock-net`. `POSTGRES_PASSWORD:?...` syntax refuses boot on missing secret (min 8 chars enforced by zod in Plan 02 config.ts). |
| T-999.2-07 | Elevation of Privilege / Path traversal | Mirror-mount defaults | mitigate | Sentinel empty paths (`/srv/devdock/apps-empty`, `/srv/devdock/claude-empty`) used when `PRODUCTION_APPS_DIR` / `CLAUDE_CONFIG_PATH` unset, preventing ambiguous bind-mount LHS. Both mounts that point at real host paths are read-only (`:ro`). Runbook (Plan 09) documents that these paths MUST be absolute. |
