# Phase 999.2: DevDock Self-Containerization - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

DevDock ships its own `docker-compose.yml` as the single source of runtime truth. Services: `app` (Next.js), `terminal` (Socket.IO), `postgres`, and `caddy`. Replaces the current systemd + host-Postgres + host-nginx-direct-to-Next setup. Auto-migrate on boot, admin seeded from env vars, zero hardcoded host paths, mirror-mounted host directories for Docker-API-driven bind mounts.

**Out of scope in this phase:**
- Actual live-VPS cutover (shipped as runbook in HUMAN-UAT.md; user runs it manually)
- npm CLI wrapper, published Docker image, install script, OSS-facing docs
- Admin UI first-run setup
- Multi-user polish (USER-01..04)
- Phase 999.1 project-native compose injection (this phase locks the contracts 999.1 plugs into, not the injection itself)

**Coexistence anchor:** Existing production routing under `/home/murx/apps` (host nginx server blocks in `/home/murx/shared/nginx`) stays untouched. Production Postgres on host stays running for production apps. DevDock self-containerization changes DevDock's own runtime only.

<decisions>
## Implementation Decisions

### Proxy & Host Coexistence
- **D-01:** Reverse proxy inside DevDock's stack is **Caddy**. Config lives in the repo, mounted into the Caddy service. Selected over keeping nginx for simpler Docker-network-by-name routing and cleaner config surface.
- **D-02:** Host nginx stays in front of everything. It owns :80/:443 and terminates TLS for `devdock.<domain>` and `*.preview.devdock.<domain>`. Host nginx proxies both to DevDock's Caddy on an internal loopback port (e.g., `127.0.0.1:8080` — exact port is Claude's discretion). This preserves the current certbot workflow and keeps production routing risk-free.
- **D-03:** DevDock's Caddy runs **HTTP-only**. No DNS-01 plugin, no cert volumes in the stack, no TLS logic inside the compose. Wildcard TLS cert acquisition stays with host certbot + host nginx.
- **D-04:** `/home/murx/apps` production routing is **untouched**. Existing `/home/murx/shared/nginx` server blocks stay as-is. No read-only mounts of host nginx configs into DevDock's proxy. Hard separation.

### Service Decomposition
- **D-05:** Two application services in compose: **`app`** (`next start` on :3000) and **`terminal`** (Socket.IO server on :3001). Matches current split; preserves independent restart and Socket.IO resource isolation.
- **D-06:** `app` service bind-mounts `/var/run/docker.sock` to call the host Docker daemon. App container runs as non-root with gid matching the host docker group (configurable `DOCKER_GID` env var / build arg). **INFRA-04 still holds**: the socket is mounted ONLY into `app` and `terminal` services, never into user-project containers.
- **D-07:** `terminal` service also bind-mounts `/var/run/docker.sock` for dockerode exec into user-project dev containers. Same security posture as `app`.
- **D-08:** **Single multi-stage Dockerfile** for the Next.js `app`. Prod target builds Next.js standalone output and runs it in a slim runtime image. Dev behavior is layered via `docker-compose.override.yml` (bind-mounts `./src` + `./server`, runs `next dev --turbopack`).

### Host Paths & Preview Network
- **D-09:** **Mirror-mount pattern** for host paths. `DEVDOCK_DATA_DIR`, `CLAUDE_CONFIG_PATH`, `PRODUCTION_APPS_DIR` are HOST paths. The compose file mounts them into `app` (and `terminal` where relevant) at **identical paths** (e.g., host `/srv/devdock/data` → container `/srv/devdock/data`). Single env var per directory works for both app code and generated compose bind-mount specs. No dual `_HOST`/`_CONTAINER` variables.
- **D-10:** Shared external Docker network **`devdock-proxy`**. Per-project compose template (`docker/templates/base-compose.yml`) declares it as `external: true`. The `dev` service attaches to BOTH `project-net` (sidecars stay isolated) and `devdock-proxy` (Caddy reaches `devdock-{slug}-dev:{preview_port}` by name). Compose generator (`src/lib/docker/compose-generator.ts`) must emit the extra network attachment.
- **D-11:** Caddy preview routing updated via **Caddy Admin API**. App POSTs/DELETEs upstream route configs on environment lifecycle events (create/start/stop/delete). Caddy admin API exposed only on the compose-internal network (never to host nginx or the internet). No file rewrites, no SIGUSR1 reloads.
- **D-12:** `CLAUDE_CONFIG_PATH` is a host path, mirror-mounted **read-only** into the `app` container (so app can validate it exists at startup and log a clear warning if misconfigured). Generated per-project compose reuses the same path string for the dev container's `:/home/dev/.claude:ro` bind mount.

### Startup Automation
- **D-13:** Migrations run via **`drizzle-kit push`** in the `app` container's entrypoint before `next start`. Idempotent, fast, no generated migration files. Me-first choice; revisit when OSS distribution phase arrives (likely switch to `drizzle-kit migrate` with committed SQL then).
- **D-14:** Admin seeded from **`ADMIN_USERNAME` + `ADMIN_PASSWORD_HASH`** env vars. Pre-hashed with bcrypt (rounds=12). Seed runs on first boot only, when the `users` table is empty. No plaintext password ever in `.env.local`. Ship a helper (`npm run hash-password` or a docs snippet) so generating the hash is trivial.
- **D-15:** Entrypoint ordering for `app`: wait for postgres healthy → `drizzle-kit push` → admin seed (idempotent guard) → `next start`. `terminal` service has no migrations; waits for postgres only if it needs DB access.

### Dev Workflow & Phase Scope
- **D-16:** Local dev is **`docker compose up`** (full stack: app + terminal + postgres + caddy). `docker-compose.override.yml` bind-mounts source and runs `next dev --turbopack` / `tsx --watch` for HMR. Dev topology matches prod — no hybrid mode, no host-native `npm run dev` path. Single canonical dev workflow.
- **D-17:** Phase deliverables: (1) root `docker-compose.yml` + `docker-compose.override.yml`, (2) `app` Dockerfile (multi-stage), (3) `terminal` Dockerfile (simpler; may share base with app), (4) Caddyfile (or Caddy JSON config), (5) updated `docker/templates/base-compose.yml` + `src/lib/docker/compose-generator.ts` for `devdock-proxy` attach, (6) entrypoint script(s) for migrations + admin seed, (7) extended `.env.example`, (8) Caddy Admin API integration in `src/lib/docker/` (new module), (9) **HUMAN-UAT.md with VPS cutover runbook** (stop systemd → adjust host nginx upstream → `docker compose up -d --build` → smoke-test → re-enable host nginx routing). Cutover itself is manual, not an executed task.
- **D-18:** Deprecate-in-place for existing deploy assets. `deploy/systemd/devdock.service` and `deploy/nginx/devdock.conf` stay in the repo with a deprecation note pointing at the new setup until the user confirms a successful VPS cutover; then they can be removed in a tiny follow-up. `deploy/nginx/devdock.conf` will need a minor edit (upstream becomes Caddy on internal port instead of Next.js :3000).

### Claude's Discretion
- Exact internal port for DevDock Caddy (host nginx forwards there)
- `DOCKER_GID` sourcing: compile-time build arg vs runtime detection in entrypoint
- Caddy Admin API auth story (binding to compose-internal network is likely sufficient; evaluate need for an admin key)
- Multi-stage Dockerfile layout (deps / builder / runner stages)
- `docker-compose.override.yml` exact structure for dev HMR
- Compose healthcheck + `depends_on` conditions (postgres readiness, Caddy admin readiness)
- Postgres version pin within `^16` family and volume naming scheme
- Bcrypt rounds + exact hash-generator helper (script vs docs snippet)
- Canonical filename: `docker-compose.yml` vs `compose.yaml` (follow whatever current tooling expects)
- Log driver and rotation settings per service
- Subnet assignments for `devdock-proxy` (avoid collisions with existing per-project networks)
- Whether `terminal` service needs DB access (inspect current terminal-server.ts for users/envs lookups)

</decisions>

<specifics>
## Specific Ideas

- **"docker compose up" is the whole story** — Both in dev and on the VPS, the user should never again need to juggle `npm run dev` + `npm run terminal:dev` + `npm run db:push` + `systemctl restart devdock`. One command up, one command down.
- **Me-first, OSS-cheap** — Every choice biased toward making the current VPS work NOW, with cheap guardrails that keep the OSS distribution path open (env-var config, no host installs, pre-hashed admin, containerized deps). No engineering effort spent on OSS polish that isn't also useful me-first.
- **Security posture preserved** — Docker socket still scoped to `app`+`terminal` only (INFRA-04). Per-project containers still publish no ports to host (INFRA-05). Caddy admin API only on compose-internal network. Host production traffic untouched.
- **Contract for Phase 999.1** — This phase locks: (a) `devdock-proxy` external network exists and project templates attach to it, (b) Caddy Admin API is how preview routes register, (c) mirror-mount semantics for host paths. Phase 999.1 (project-native compose injection) builds on those contracts.
- **Hash-password helper** — User asked about generating `ADMIN_PASSWORD_HASH`. A small `npm run hash-password` script (or a one-line docs snippet) lets the user generate a bcrypt hash locally before populating `.env.local`. Trivial DX improvement.

</specifics>

<deferred>
## Deferred Ideas

- **Caddy auto-TLS with DNS-01** — Moving TLS into the compose stack (Caddy gets its own wildcard cert via DNS provider). Deferred because host nginx already handles certs and moving it introduces provider-API dependency. Revisit in the OSS distribution phase (999.3+) when users don't have a host nginx.
- **Live VPS cutover execution** — This phase provides the runbook; running it is a separate manual step the user performs when ready. Could be a tiny follow-up phase if structured execution is wanted.
- **Removing deprecated `deploy/systemd/devdock.service` + simplifying `deploy/nginx/*.conf`** — Tiny cleanup phase after successful cutover confirmation.
- **Generated SQL migrations (drizzle-kit migrate)** — Safer for schema evolution at scale. Deferred to OSS distribution phase when multi-environment upgrades matter.
- **Admin UI first-run setup** — Replace `ADMIN_PASSWORD_HASH` env var with a one-time web form on first visit. Nicer OSS onboarding; me-first doesn't need it.
- **npm CLI wrapper and published Docker image** — Explicit 999.3+ scope per roadmap entry.
- **Secrets management beyond `.env.local`** — Docker secrets, sops, vault integrations. Overkill for single-user VPS; revisit for multi-user/OSS phases.
- **Backup/restore runbook for the new postgres volume** — Worth a small ops doc; defer to a polish phase.

</deferred>
