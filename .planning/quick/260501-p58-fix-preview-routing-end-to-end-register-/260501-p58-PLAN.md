# Quick Task 260501-p58: Fix preview routing end-to-end

## Problem (3 independent bugs)

**Bug 1 — `PREVIEW_DOMAIN` config has the wrong port.**
`.env.local:64` and `.env:64` set `PREVIEW_DOMAIN=127.0.0.1.nip.io:3000`. Caddy is the only published port: `127.0.0.1:8080` (per `docker-compose.yml`). The Preview button builds URLs as `http://{envId}.{NEXT_PUBLIC_PREVIEW_DOMAIN}/` — those hit port 3000 which isn't published. Stale config from before Phase 999.2 self-containerization (when DevDock ran on host 3000 directly).

**Bug 2 — Auth canonical-host redirect catches preview subdomains.**
`Caddyfile` (added by quick task 260501-ihv) 308-redirects every request whose Host != `localhost:8080`. Preview subdomains (`{envId}.127.0.0.1.nip.io:8080`) trip this redirect first and never reach the per-env reverse_proxy route. The redirect was scoped narrowly enough to fix auth cookie host scoping, but it also catches preview subdomains.

**Bug 3 — Caddy preview route only registered on env start, never on PATCH.**
`src/lib/docker/caddy-lifecycle.ts:24` no-ops when `env.previewPort` is null, and the call site is `src/app/api/environments/[id]/start/route.ts:8` only. When a user sets `previewPort` later via Edit Environment (PATCH), no Caddy route gets registered until the next stop+start cycle. With 260501-obf and 260501-ogb landed, this becomes the dominant flow — fresh envs ship with port 3000, but legacy envs upgraded via Edit Save still need a Caddy route.

## Plan 01: PATCH handler registers/deregisters preview route on running envs

**Files:** `src/app/api/environments/[id]/route.ts`

**Action:**
After the DB update at line 174, if the env was running BEFORE the update (so we know its previewPort change is observable):
- If `previewPort` changed and new value is non-null: call `registerPreviewRoute({ id, slug, previewPort })` (which is idempotent — the addPreviewRoute helper already does best-effort delete+POST).
- If `previewPort` changed to null: call `deregisterPreviewRoute(slug)`.
- If env is not running, skip — `start` will register on next start.

Existing imports already cover `caddy-lifecycle`. Wrap in try/catch following the same swallow-and-log pattern the lifecycle module uses internally.

**Verify:**
- PATCH portfolio-retest with previewPort=3000 (already running) → curl Caddy admin `/id/preview-portfolio-retest` returns the registered route.
- PATCH same env with previewPort=null → curl admin returns 404.
- PATCH with no previewPort change (e.g. only name change) → no Caddy admin calls.

**Done when:**
- PATCH on running env with previewPort change updates Caddy admin
- Stop+start lifecycle still works unchanged
- No regression on PATCH for stopped envs

## Plan 02: PREVIEW_DOMAIN port → 8080

**Files:** `.env`, `.env.local`, `.env.example`

**Action:**
Change `PREVIEW_DOMAIN=127.0.0.1.nip.io:3000` → `PREVIEW_DOMAIN=127.0.0.1.nip.io:8080` and same for `NEXT_PUBLIC_PREVIEW_DOMAIN`. Update the inline comment to mention the matched Caddy port. Apply to all three files for parity.

**Verify:**
- `grep PREVIEW_DOMAIN .env*` shows `:8080` everywhere
- After app rebuild, `getPreviewUrl(envId)` (env-card.tsx) returns `http://{envId}.127.0.0.1.nip.io:8080/`

**Done when:**
- All three .env files updated
- App rebuilt so `NEXT_PUBLIC_PREVIEW_DOMAIN` is baked at the new value

## Plan 03: Caddy canonical-host exemption for preview subdomains

**Files:** `Caddyfile`, `docker-compose.yml`

**Action:**
1. `docker-compose.yml` caddy service: add `PREVIEW_HOST: ${PREVIEW_HOST:-127.0.0.1.nip.io}` to the `environment` block. This is the *host portion only* (no port) of `PREVIEW_DOMAIN`, used purely for Caddyfile config substitution.
2. `Caddyfile` `@noncanonical` matcher: add a second `not header_regexp Host (?i)\.{$PREVIEW_HOST}(:\d+)?$` clause so preview subdomains skip the canonical-host redirect. Matcher conditions are AND-combined in Caddy, so adding a `not` clause means: redirect only if Host is neither the canonical host NOR a preview subdomain.

Note on regex: dots in `127.0.0.1.nip.io` are regex meta. Caddy's `header_regexp` uses Go's regexp syntax. The substituted value will contain literal dots; we deliberately don't escape them because nip.io subdomains share that pattern and treating dots as `.` (any char) is safe in this match — false positives would have to be Hosts ending in something like `127X0X0X1XnipXio` which doesn't occur.

**Verify:**
- Caddy reloads cleanly after Caddyfile change (config-adapt errors surface in logs).
- `curl -H "Host: {envId}.127.0.0.1.nip.io:8080" http://localhost:8080/` no longer returns 308 — instead returns whatever the per-env upstream returns (200 if server is up, 502 if not).
- `curl -H "Host: 127.0.0.1:8080" http://localhost:8080/` still returns 308 → `localhost:8080` (auth flow regression check).

**Done when:**
- `docker compose up -d caddy` picks up the change (with Caddy config reload)
- Preview URL reaches the per-env upstream
- Non-preview non-canonical hosts still get redirected

## Out of scope

- Backfilling Caddy routes for envs that started before this fix (will get registered on next start, or via PATCH after the user touches Edit). Acceptable since these envs are short-lived dev environments.
- TLS / HTTPS for preview subdomains — D-03 says host nginx terminates TLS in production; local dev uses HTTP only.
- Removing the deprecated `/api/preview/[[...path]]` Next.js fallback route — D-18 says that's a follow-up phase.
- Auto-cleanup of Caddy routes when an env is deleted — already handled by the existing stop/delete lifecycle.
